Trade-Finding Hub

Mockup options for feedback #156 — one place to choose how you find trades, and the home for every targeting preference.
Design mockups only · Chalkline dark palette · ~390px frame · no app code changed

The operator's vision: give users an easy, seamless way to pick a trade-finding mode — Fully Guided, Specific Team, Specific Player (trade for / away / both), or Manual Calculator — and make this same surface the home for untouchables, positions-needed, positions-to-trade-away, and named players to acquire/shop. Inspiration: NBA2K's find-a-trade board (pick players for/away/both → engine returns constrained suggestions). All three variants build on features FTF already ships.

What's real vs. what needs building

Every variant is mostly a re-composition of shipped features. The honest gaps:

Data & feature dependencies

  • 4 modes — Guided, Player, and Calculator all exist today (trade generation + FB-47 targeting + TradeCalculatorScreen). have
  • Specific Player · for / away / both — the FB-47 finder already supports pinned_give / pinned_receive with a direction toggle; "both" is expressible now. The mockups just make it the visual centerpiece. have
  • Untouchables — asset_prefs list_type untouchable (feedback #95) is live. have
  • Named players to acquire / shop — asset_prefs target + FB-47 pins cover this. have
  • Positions-needed / positions-to-trade-away as manual chips — already in OutlookSheet (acquire_positions / trade_away_positions). have
  • Recommendation chips on those positions ("Thin at WR", "Deep at RB") — the roster analysis exists (analyze_roster_strengths returns position_needs + position_surplus), but today only inferred_outlook is exposed to clients. Surfacing needs/surplus needs a thin endpoint (or extra fields on the prefs response). data exists wiring new
  • Specific Team mode (scope generation to one league-mate) — the engine already iterates opponents and rosters are available; a single-opponent scope is a small backend param + picker. small new
  • The hub / drawer / lens shell itself — net-new client structure; no new backend. new UI

Recommendation

Go with Variant B — Launcher Hub

  • Best fit for the vision's two jobs. The hub cleanly separates "pick a mode" (four cards) from "set your preferences" (Trade DNA panel), so both feel first-class instead of one hiding under the other.
  • The NBA2K board gets the room it deserves. Specific Player as a full-screen two-column FOR/AWAY board is the closest adaptation of the 2K pattern and reads well on a phone — cramped in A and C.
  • Seamless jumping is preserved via the lateral quick-switch chip row inside every mode, so the one-extra-tap cost of a hub mostly disappears in practice.
  • Card-launcher pattern already exists in the app (RankHomeScreen's method chooser), so it's familiar to build and to users.

If the priority is minimal build: Variant A ships fastest (mode tabs are today's trades subnav widened, prefs drawer is a re-home of existing sheets). If the priority is a power-user "cockpit": Variant C is the most seamless but should wait until the recommendation-chip endpoint and lens-emphasis interactions are worth the density.

Open questions for the operator

  • Should the recommendation-chip endpoint be built now (unlocks the "rec" chips in all three), or do we ship with manual position chips first and add recs in a fast-follow?
  • Is "Specific Team" worth the small backend scope work for v1, or should it be a fast-follow after the other three modes land?
  • Do preferences live per-league (matches asset_prefs / league_preferences today) — confirm the hub is always league-scoped, with a visible league pill as mocked.
