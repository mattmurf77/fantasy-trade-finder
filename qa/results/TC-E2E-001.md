### Objective
Validate the full mobile-client journey through a real local server — the layer
*wiring* (in-memory sessions, async job queue, write-through DB persistence,
two-party match detection) that the unit suite never exercises — with mobile
timeout budgets as the pass bar.

### Scope
- **In scope:** session_init (DB-roster path), trio→rank3 loop with Elo
  persistence, async trade generation + card quality, swipe→mirror→match (both
  the likes_you instant path and the two-session two-step path), disposition
  lifecycle incl. 409/404/400, post-run DB integrity, per-call timing budgets.
- **Out of scope:** Sleeper passthrough login (covered by planned TC-INT cycle);
  exact K-factor Elo math (unit-tested in `test_elo_memoization.py`); sf_tep
  format variant and restart-resilience (planned variants, see below); Postgres.

### Preconditions / Setup
- Scratch copy of `data/trade_finder.db` (live DB never written); fresh copy per run.
- Local Flask via `DATABASE_URL=sqlite:///qa/e2e/scratch/trade_finder_e2e.db`, port 5099.
- Primary user `test_user_fp_1` in `test_league_lakeview` (12 members, 646
  member_rankings each for both fixture users); counterparty chosen dynamically
  from whichever member the top fresh card targets.

### Inputs / Steps
Automated harness: [qa/e2e/tc_e2e_001.py](../e2e/tc_e2e_001.py) — run with
`python3 qa/e2e/tc_e2e_001.py`. Stages: (1) session_init + ping, (2) 3× trio→rank3
on RB, (3) generate + poll + card-quality battery + cache-hit probe, (4a) like a
`likes_you` card → instant match, (4b) like a fresh 1-for-1 card → counterparty
session likes the mirror via FB-46 card-context echo → match, (5) disposition:
accept → still pending → second accept → accepted → 409 repeat / 404 unknown /
400 bad decision, (6) integrity sweep + debug-log scan + budget audit.

### Expected Result
All checks pass: contract shapes, card quality invariants (fairness ∈ [0,1],
basis enum, rosters own the players traded, unique ids), 3 swipe rows per rank3,
atomic member_rankings republish, match only when mirrored, status rolls only
when both decide, no new orphans/enum violations, all calls within mobile budget.

### Actual Result
**67/67 PASS.** Machine-readable evidence: `qa/e2e/scratch/TC-E2E-001-run.json`.
Highlights: 31 cards in 0.8–1.5 s (budget 30 s); cache-hit second generate in
2–4 ms; cold boot ~1–2 s; session_init 370–590 ms warm; slowest call overall was
the first rank3 at ~770 ms (first-touch cache build), subsequent 11–14 ms.
Two harness iterations were themselves findings: deck is *intentionally* not
composite-sorted (thompson_deck on), and a likes_you card matches on first like
(correct Tier 2.3a behavior).

### Observations & feedback (no change required)
- **Orphaned league_members are "members who never logged in":** counterparty's
  first session_init upserted their users row and the orphan count *fell* 41→38.
  The 41-orphan recon finding is mostly benign by design; worth one line in
  docs/data-dictionary.md rather than a fix.
- **Deck ordering contract is flag-dependent** (composite-sorted only when
  `trade.thompson_deck` off). Nothing in docs/api-reference.md says deck order is
  stochastic; a client assuming sorted order would be wrong. Suggest a sentence there.
- **FB-46 card-context reconstruction works as designed** and is load-bearing for
  match detection across sessions — good resilience pattern, now regression-covered.
- **Generate→swipe field asymmetry:** cards serialize `give`/`receive` as player
  objects but swipe expects `give_player_ids` arrays. Fine (mobile maps it), but
  worth documenting as the canonical echo contract.
- Timing posture overall is healthy at this data scale; the real perf risks
  remain the ones from recon (cold Sleeper fetch in session_init on prod, v3
  enumeration without budgets) which this small league doesn't exercise.
