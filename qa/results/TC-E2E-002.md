### Objective
Confirm that losing in-memory state on a deploy/restart degrades gracefully —
clean 401/404 rather than hangs or 500s — and that the FB-46 card-context echo
keeps swiping functional against a deck whose in-memory entry is gone.

### Scope
- **In scope:** stale session-token rejection; stale job-id eviction (no hang);
  data survival across restart; FB-46 swipe of a pre-restart card; full
  functionality of a re-initialized session.
- **Out of scope:** in-flight job *resumption* (jobs are intentionally not
  durable — this verifies graceful loss, not recovery); concurrent-restart races.

### Inputs / Steps
Automated: [qa/e2e/tc_e2e_002.py](../e2e/tc_e2e_002.py). Boot → session + generate
(capture token, job_id, a 1-for-1 card) → **terminate and boot a fresh process
on the same DB** → assert old token 401s, old job 404s within <5s, data
survived, FB-46 swipe of the pre-restart card succeeds and persists, and a
fresh generate works.

### Expected Result
Old token → 401; old job_id → 404 with no hang; `member_rankings` survive;
pre-restart card swipe → 200 (reconstructed) and `trade_decisions` +1; new
session generates a fresh deck.

### Actual Result
**9/9 PASS.** Old token → 401; old job → 404 in 0.00s; 646 `member_rankings`
rows survived; FB-46 swipe reconstructed + persisted (+1 decision); post-restart
generation produced 31 cards. Evidence: `qa/e2e/scratch/TC-E2E-002-run.json`.

### Outcome
**PASS** — the in-memory job/session loss is a graceful degradation, not a
failure mode. The recon operability concern is closed: clients get clean,
actionable errors (re-init on 401, re-tap on 404) and swiping survives deploys.

### Observations & feedback (no change required)
- **FB-46 is load-bearing for deploy UX** — without the echoed give/receive
  context, every mid-session deploy would turn the user's next swipe into an
  "Unknown trade_id" error. Now regression-covered across an actual restart.
- **The 404-on-stale-job is instant** (0.00s) because the lookup is a dict miss,
  not a DB round-trip — good; a client polling through a deploy gets a fast,
  unambiguous "regenerate" signal rather than a timeout.
- **Write-through persistence is what makes this safe**: rankings, decisions,
  and matches are all in the DB before the response returns, so only *ephemeral*
  state (the deck ordering + job progress) is lost — and that's cheap to rebuild.
- Pairs naturally with TC-E2E-001: that proved the happy path, this proves the
  deploy-interrupted path. Together they cover the realistic production lifecycle.
