# Glossary

Domain terms used throughout the codebase. Add a term when new jargon appears.

---

**Elo** — Rating system from chess. Each player has a numeric rating; comparing two players updates both based on actual vs. expected outcome. Used to rank fantasy players within a single user's preferences.

**K-factor** — How aggressively an Elo update moves a rating. Higher K = bigger swings. Per decision type (defaults from `model_config`):
- Rank (3-player matchup): `elo_k = 32`
- Trade like: `trade_k_like = 8`
- Trade pass: `trade_k_pass = 4`
- Trade accept: `trade_k_accept = 20`
- Trade decline correction: `trade_k_decline_correction = 20`

**3-player matchup (trio)** — User orders 3 players best→worst. Decomposed into 3 pairwise rows in `swipe_decisions` (A>B, B>C, A>C) → 3 Elo updates. ~2.6× more information per interaction than pairwise.

**KTC value** — Dynasty trade value styled after [KeepTradeCut](https://keeptradecut.com/). Used as UI display value and as the seed for initial Elo. Curve params in `model_config`: `ktc_max=10000`, `ktc_k=0.0126`, `ktc_fallback_rank=300`.

**DynastyProcess (DP)** — GitHub CSV of consensus dynasty values; the spine of the seed source for initial Elo via `data_loader.py` (owns the universe, crosswalk, and 0–10k value scale). Since #145 blended with KeepTradeCut — see **Blended consensus**.

**KeepTradeCut (KTC)** — Second dynasty-value source, blended into the DP baseline since #145 (`data_loader.py`). No official API: its `dynasty-rankings` page embeds the full player list as a `var playersArray = [...]` HTML literal, scraped once per boot (fail-soft → DP-only). Distinct from the older **KTC value** curve entry above, which is FTF's own KTC-*styled* rank→value curve in `model_config` — unrelated to this real-KTC blend.

**Blended consensus** — The baseline consensus seed values, a weighted blend of DynastyProcess and KeepTradeCut computed at pool build (`data_loader._apply_consensus_blend`). Design: DP is the spine (universe/crosswalk/value-scale); KTC values are **rank-normalized onto the DP value curve** per format, then averaged `(1−w)·dp + w·ktc` with `w = model_config.ktc_blend_weight` (default 0.5). This imports KTC's ordering opinion while keeping the value distribution DP-shaped, so tier occupancy and the #117 affine Elo calibration (top asset ≈ 4 firsts) are preserved. The pool universe is unchanged — only players already DP-derived are blended (position-strict match per #127); unmatched KTC players are ignored, unmatched pool players keep pure DP. `sf_tep` TE seeds then get the `tep_te_uplift` premium (#148 — DP's `value_2qb` is premium-less plain superflex). Kill switch: `ktc_blend_weight = 0` (+ `tep_te_uplift = 1`) reproduces the pre-#145 DP-only seeds byte-for-byte. See [config-reference](config-reference.md) + [runbook](runbook.md).

**Tier band** — Bucket of similar Elo ratings rendered as a labeled, colored badge. Since 2026-07-12 (#117) the taxonomy is the **8-tier pick-value ladder**: `firsts_4plus` ("4+ 1sts", red) / `firsts_3` ("3 1sts", fuchsia) / `firsts_2` ("2 1sts", gold) / `first_1` ("1st", teal) / `second` ("2nd", sky) / `third` ("3rd", pink) / `fourth` ("4th", lime) / `waivers` ("FA", gray) — each floor a rung of the anchor/pick Elo ladder, so a tier name states what a player in it is worth in draft-pick terms. Bands are position- and format-uniform in Elo space (pick value is position-uniform); the consensus seed recalibration shipped with #117 (`data_loader.seed_elo_for_value`) lets top consensus assets actually reach the multi-first tiers (top asset ≈ 4 firsts). Retired: the abstract Elite/Starter/Solid/Depth ladder (2026-07-11) and the interim six-tier `firsts_2plus`/`bench` keys (2026-07-12). Canonical bounds in `backend/tier_config.json`; keys/colors in `docs/cross-client-invariants.md`.

**FA (tier)** — Display label of the bottom tier (key `waivers`, gray): below-4th-round value, i.e. waiver-wire / free-agent fodder. Renamed from "Waivers" on 2026-07-17 (label-only — the `waivers` key, gray hex, and Elo band are unchanged). Distinct from the "FA" team fallback shown on players without an NFL team.

**Base first** — The unit of the pick-denominated features (2026-07-10, adopted from the TI-CALC teardown, [docs/competitor-teardown-ti-calc.md](competitor-teardown-ti-calc.md)): a **generic Mid 1st Round Pick**, Elo seed 1650 in `GENERIC_PICK_SEEDS` (`backend/server.py`), ≈ 2117 in value space. "Worth 2 firsts" = 2 × value(base first).

**Pick anchor** — An explicit user statement of a player's worth in draft capital ("worth 2 firsts", "worth a mid 2nd", "no trade value"), made in the mobile Pick Anchor wizard (`PickAnchorScreen`, reached from the Rank tab's menu, the Build-your-board chooser, or the `anchor` launch preference — the Tiers-header link was removed in 1.5.4 #99) and saved via `POST /api/anchor/save`. Pins the player's Elo as an authoritative override — the same mechanism as a tier drag. Anchor VALUES are position-uniform by design (the pick ladder drives uniform valuation across position groups); since the 2026-07-11 pick-value tier ladder the bands are position-uniform too, so every anchor lands in the tier that carries its name. The explicit-user-statement sibling of the trio calibration plan's Lever B (ordinal anchor placement).

**Pick-gap equivalent** — The `gap` field on `/api/trade/evaluate`: the consensus package delta re-expressed as draft capital — `firsts` (units of the base first) plus the nearest single generic pick — so a verdict becomes an actionable counteroffer ("ask for ≈ a Mid 2nd back"). Rendered in the calculator's `ConsensusVerdictCard`.

**Pick sublabel** — RETIRED (2026-07-11). The 1.5.4 #103 display-only draft-pick-terms sublabel beside tier names (`Elite 1st+`), computed by `mobile/src/utils/pickTerms.ts`. Superseded by the pick-value tier ladder: tier labels ARE pick terms now, so the helper and sublabels were removed.

**Pick-value scale** — 1.5.4 #111 (re-derived 2026-07-12 for #117): a user's declaration of how many firsts a top-tier dynasty asset is worth (`top_tier_firsts`: 2/3/4, default 4 = the consensus math since the #117 seed recalibration). Set from the pill row at the foot of the Pick Anchor wizard, persisted per user + scoring format (`users.anchor_scale`, `GET/POST /api/anchor/scale`). Re-spaces only the wizard's multi-first anchors along a power curve (γ = log 4 / log N) so the user's own "N firsts" answer pins to the default top-tier Elo (≈ 1927) while "1 first" stays the actual generic Mid 1st (Elo 1650); every consensus-denominated surface (pick pool assets, calculator pick-gap line, the tier-ladder band floors) is unaffected — see `docs/cross-client-invariants.md` § Pick anchor keys.

**Quick set (tiers)** — 1.5.4 #104: the guided tier-assignment walk (`QuickSetTiersScreen`). Since #119 (2026-07-12) a first-class ranking method — entered from the Tiers header, the rank-home chooser (lowest-effort card, "recommended" tag), the Rank tab's action sheet, or directly at launch via `rankingMethodPref: 'quickset'`. Per position, presents one tier at a time from the top of the ladder ("4+ 1sts") down to FA (8 steps since #117) with a tappable grid of small player cards (name + team + age + current tier — #140 replaced the redundant position token with team + age on these position-scoped grids); saving a tier commits just that tier through `POST /api/tiers/save` (composes with the existing board — only submitted pids change; re-visited deselections travel as `cleared_pids`) and advances. Players claimed by an earlier tier drop from later grids. Finishing the walk offers **Quick rank** (below) as the natural next step.

**Quick rank** — #136 (2026-07-12): the within-tier ordering pass after Quick set (`QuickRankScreen`, Rank-stack route `QuickRank`). Same guided construction (position tabs, tier-by-tier walk down the ladder, per-tier save) but the grid shows every player already IN the current tier: tapping stamps the next rank number on the chip (click order; tap again to unclick and renumber), and Save posts the tier's players to `POST /api/rankings/reorder` — clicked order first, unclicked players appended in their current order (they become ranks N+1…end). `apply_reorder` permutes the Elo multiset of exactly the submitted ids (subset-safe v1.7.0 permutation semantics), so a Quick rank save can never move a player across a tier boundary or touch the rest of the board. Tiers with fewer than 2 players are skipped by construction. Entered from the Quick set finish prompt ("Rank within your tiers?") or the Rank tab's action sheet; deliberately NOT a launch-routable `rankingMethodPref` (#122, unselected).

**Value-aware copy (`value_rank` mapping)** — 2026-07-17, #124/#139: the cross-format board copy semantics of `POST /api/tiers/copy-from-format`. Because tier labels are pick-denominated ("worth 4+ 1sts") and the two formats' consensus value curves differ (QBs most of all), tier labels do NOT carry across formats — a player worth 4+ firsts in SF is not worth 4+ firsts in 1QB. The copy therefore preserves the user's per-position **rank order** from the source board and re-seeds each player's Elo from the **target format's consensus seed curve** at that rank (`RankingService.apply_value_map` — `apply_reorder`'s permutation trick pointed at the target seed distribution, keeping the convex value curve intact so target tier occupancy matches target consensus occupancy). QBs shift most between SF and 1QB, by design; the mobile confirm dialog says so before firing. Replaces the retired label-preserving copy (which mispriced QBs in both directions).

**Edited trade / suggested swaps** — Feedback #86 (mobile swipe deck): tapping the swap affordance next to any player on a suggested trade card opens `SwapPlayerSheet`, which offers **suggested swaps** (that roster's players closest in consensus value to the outgoing player — a "keeps it fair" shortlist) above the full roster grouped QB→RB→WR→TE. The modified card is an **edited trade**: it shows an `EDITED` badge, is re-priced live via `/api/trade/evaluate` Mode B, and carries a derived `trade_id` (`<original>::edited`) so a like/flag/Send-in-Sleeper records the MODIFIED package — the unknown id deliberately routes the swipe through the server's FB-46 context-reconstruction path instead of the in-memory original card.

**Team outlook** — User's strategic mode for a league (`league_preferences.team_outlook`): `championship`, `contender`, `rebuilder`, `jets`, `not_sure`. Since the trade-engine v2 rebuild, outlook feeds the **outlook blend** (see below) — the old post-hoc score multiplier (`team_outlook_multiplier`) is deleted, and the legacy engine path ignores outlook entirely. The historic multiplier keys (`boost_strong`, `vet_age`, …) still exist in `model_config` but are unused.

**Package weights / diminishing returns** — Multi-player trade sides apply diminishing weights so "5 bench guys for an elite WR" doesn't look equal. From `model_config`: `package_weight_1..5 = 1.00, 0.75, 0.55, 0.40, 0.28`.

**Positional preferences** — The user's `acquire_positions` / `trade_away_positions` for a league. Since the v2 rebuild these are a **hard filter** on candidate packages (a card must receive an acquire position / give a trade-away position when set) in both engine paths; the old soft multipliers (`pos_acquire_bonus` etc.) are deleted from code though their `model_config` keys remain.

**Value space / `elo_to_value`** — The v2 engine does ALL trade math in dynasty-value units, not Elo. `elo_to_value(elo) = elo_value_base · exp(elo_value_k · (elo − elo_value_ref))` maps each side's (shrunk) Elos onto the same scale as consensus KTC-style values, so surpluses and fairness are commensurable. `backend/trade_service.py:235`.

**Marginal value (over-replacement)** — Tier 2 valuation (`trade.marginal_value`): a player's worth to a *specific roster* is `max(0, value − replacement_level(position))` plus a `bench_credit_rate` (15%) credit, where the replacement level is that roster's best non-starter at the position (waiver baseline if the position is thin). Makes clogger packages collapse and need-fillers keep value.

**Now/future value & outlook blend** — Tier 2 (`trade.outlook_blend`): every player has a win-now and a long-horizon age multiplier (per-position curves `_AGE_NOW_CURVE` / `_AGE_FUTURE_CURVE`); the user's outlook sets α (`outlook_alpha_*`, championship 1.00 → jets 0.10) and the value used is `α·now + (1−α)·future`. An *input* to surplus math, so it composes with the fairness gate — unlike the deleted post-hoc multiplier.

**Inferred outlook** — Backlog #1 (`trade.outlook_infer`): the opponent equivalent of the above. Where the user's α comes from their declared outlook, each *opponent's* α is resolved per trade as declared (`league_preferences`) → **inferred** → `not_sure`. `infer_team_outlook` classifies a team's contend/rebuild window from roster shape — veteran value share, youth value share, and draft-pick-capital share — bucketing into contender / not_sure / rebuilder (the extremes championship/jets are reserved for self-declaration). The opponent's side of every candidate trade is then priced through their α, so the engine stops offering aging vets to rebuilders. Cards carry `match_context.opponent_outlook = {value, source}`.

**Positional need fit** — FB-96 (`trade.need_fit`, feedback #96): an automatic per-card score in [0,1] (0.5 = neutral) computed from the two rosters' positional-strength profiles alone — no user input. Each traded QB/RB/WR/TE contributes: giving from the user's deep position into the opponent's thin one scores high, and receiving at the user's thin position from the opponent's deep one scores high ("you're strong at WR, they're strong at RB — swap"). Superflex raises the QB "loaded" bar by one body. Applied as a bounded composite multiplier (`1 + need_fit_weight·(fit − 0.5)`) after all gates, so it reorders acceptable trades without rescuing gated ones. Serialized on cards as `need_fit`. Distinct from FB-47's `partner_fit`, which ranks *opponents* against the user's explicitly stated targets.

**Range-overlap fairness** — v2 fairness gate: each package's consensus value gets an uncertainty half-width from comparison counts (`range_base/√(1+n)`, value-weighted); a trade passes when the two sides' value intervals overlap OR the point ratio clears `fairness_threshold`. High-uncertainty players (rookies) pass more easily. The serialized `fairness_score` stays the point ratio (0–1).

**Consensus-basis card** — v2 card generated for an opponent with NO real rankings (`basis="consensus"`): divergence math against fabricated Elos is noise, so the engine surfaces simple fair-by-consensus, roster-fit-oriented ideas instead. Clients label them "Fair-value idea". Divergence cards carry `basis="divergence"`.

**Consensus basis (scoring)** — Pricing a trade purely on market-wide DynastyProcess consensus values (`elo_to_value` over the universal-pool seed), with NO personal Elo, NO league, and NO outlook blend. The bottom layer shared by every value surface; the open trade calculator (#27) scores entirely on this basis, while the logged-in engine layers personal Elo and league awareness on top.

**Consensus positional rank** — FB4-61 (Tiers tile stats): a player's 1-based rank *within their position* by consensus seed value over the active format's universal pool — the market twin of the user's positional rank, so the tile strip shows the same two stats for both sides (`You #4 ▲2 30d · Cons #7 ▼1 30d`). Computed in `trends_service.compute_consensus_pos_ranks`, memoised per (format, day), and serialized on `/api/rankings` as `consensus_pos_rank`; its 30d trend (`consensus_pos_rank_delta_30d`, positive = up) compares against the oldest prior-day `player_value_history` snapshot in the window and is omitted until that history exists.

**Open trade calculator** — Backlog #27: the public, no-login `web/calculator.html` page + its `POST /api/calc/score` / `GET /api/calc/values` endpoints (flag `calc.open_calculator`). Scores any two asset lists on the consensus basis and renders a #6 verdict, ending on a "connect Sleeper" conversion CTA. The acquisition/SEO front door (the category head term "dynasty trade calculator"); the consensus-only sibling of the session-authed rescore endpoint.

**Manual Trade Calculator (mobile)** — `TradeCalculatorScreen` in the mobile Trades stack (Calculator pill on the Trades tab). Two modes: **Real values** (default) prices any hand-built trade on live consensus values via the public `POST /api/trade/evaluate` + `GET /api/trade/values` endpoints (server-authoritative — reuses the finder's `_fairness_v3`, per `docs/plans/manual-trade-calculator-plan.md`); **Demo league** runs on seeded mock data (`mobile/src/data/tradeCalcMock.ts` + `utils/tradeCalcMath.ts`) demonstrating the future league-aware dual-**board** version — partner tendencies, arbitrage badges (Target / Sell high), draft picks priced by board bias. Both modes: fair-package + balance-the-trade suggestions, persisted draft (`ftf:tradecalc:v1`), native share. Sibling of the staged **Open trade calculator** web tool (backlog #27) — consolidate contracts when that lands.

**Board** — In the calculator UI, one owner's personal ranking set: their value for every player ("Your board" vs "Their board"). A trade reads as agreeable only when both boards like it — the same mutual-gain rule the finder uses.

**Likes-you card** — Tier 2 (`trade.likes_you`): a card whose mirror a league-mate already liked in the last 90 days (and which is still roster-valid). Flagged `likes_you: true`, boosted/pinned to the top of the deck (max 3 injections), rendered with the "👀 They're interested" pill.

**Bad-trade flag** — Feedback #85: the "Bad trade?" tertiary action under the TradesHome swipe deck. Distinct from a **pass** (not interested — an ELO signal): a flag means "the engine got this one wrong" and writes a `bad_trade_flags` row (package + counterparty + engine telemetry snapshot, `POST /api/trades/flag`) for operator review via `GET /api/trades/flags/admin`, feeding iteration on the trade-generation logic. Flagging also advances the deck like a pass (a flagged trade is implicitly not interesting). One flag per (user, league, give set, receive set); carries no ELO signal itself.

**Sweetener** — Tier 3 (`trade_engine.v3`): a low-value asset added to the under-paying side of a *near-miss-fair* trade to bring it into the fairness band. Serialized as `sweetener: {player_id, side}` (the player is already in give/receive); clients render "+ {name} added to balance the deal".

**Thompson deck ordering** — Tier 2 amendment A5 (`trade.thompson_deck`): instead of a learned acceptance model (no training data yet — ~20 labels), the deck order is exploration-randomized by drawing one Beta(1+likes, 2+passes) sample per card *shape* (`1x1`, `2x1`, …) from the user's own decision history and multiplying ordering keys by a bounded (0.5, 1.5) factor. Deterministically seeded per job.

**Deck diversification** — Tier 2 amendment A6 (`trade.deck_diversity`): a player can only be traded once, so one stud saturating every member's deck caps total possible matches. Cards whose top receive asset appeared in ≥ `diversity_user_cap` other members' recent decks get a `diversity_penalty` ordering multiplier, and the served deck keeps ≤ `deck_max_per_target` cards per target (never below 5 cards, never dropping likes-you cards).

**Three-team cycle** — Tier 3 (`trade.three_team`): a kidney-exchange-style cycle A→B→C→A where every team's net marginal surplus clears a bar (`cycle_min_net`); found by clearing a directed gain graph over the league. A distinct card type; all three members must agree.

**Trade scoring** — Composite of mismatch and fairness components, weighted `mismatch_weight=0.70` / `fairness_weight=0.30` (both paths). Legacy path: cards below `min_mismatch_score=40` or above `max_value_ratio=2.5` are filtered; `max_candidates=30` per opponent before sort. v2 path: the mismatch term is the capped harmonic mean of the two sides' surpluses (see *Value space*). Both paths reject cross-side Elo gap > `trade_elo_gap_max=250`.

**Trade-math adjustments (flag-gated)** — Optional penalties enabled via feature flags:
- `qb_tax_rate=0.075` — penalty for receiving a premium QB without giving one back
- `star_tax_per_tier_gap=0.10` × `star_tax_elite_multiplier=1.5` for Tier-1 stars
- `roster_spot_penalty=0.05` per extra roster spot used
- `roster_clogger_penalty=0.10` additional per player beyond 2 in 3+ one-way trades, threshold `roster_clogger_threshold=3`

**Tier engine** — Pre-unlock matchup filter that focuses trios on top `tier_size=24` per position. Post-unlock, mixes in lower-tier players at probability `mix_in_rate_base=0.35` rising to `mix_in_rate_max=0.80` as comparisons saturate (`mix_in_saturation_pct=0.70`). Pre-unlock mix-in begins at `mix_in_pre_unlock_start=5` interactions. Toggle via `tier_engine_enabled` (1.0).

**Cross-position trio** — #132 trio variety: 3 same-tier players from ≥2 different positions (`ranking_service._cross_position_trio`, reasoning "Cross-position tier check: <tier>"). Probes how the user values equally-tiered assets across position groups — the comparison the cross-position trade finder relies on. Served only post-unlock (all four positional interaction thresholds met) at a small share (`trio_cross_pos_rate=0.15`, out of the tightest remainder); own tier cursor + the standard anti-repeat. Distinct from the **All board** (mobile Tiers "All" tab, also #132): the merged cross-position tier view whose saves are routed per position client-side.

**Smart matchup** — Claude-powered matchup selection. Generates ~10 candidate pairs (nearby Elo, not yet compared), asks Claude to pick the most dynasty-informative. Toggle via `smart_matchup_enabled` (1.0). Falls back to algorithmic without `ANTHROPIC_API_KEY`.

**Mutual gain trade / Trade match** — Both sides improve by their own Elo math. When both users like mirrored trades, a `trade_matches` row is created (status `pending`) and both inboxes get a `trade_match` notification. Status is a *disposition* state (`pending` → `accepted`/`declined` once both parties decide), not a separate kind of trade — user-facing surfaces bucket by segment (mutual vs awaiting), never by status (#91).

**Awaiting them** — A trade the user liked whose counterparty hasn't liked the mirror yet: a `trade_decisions` like with no `trade_matches` row for the same league + player sets. Second segment on the Matches screen (`/api/trades/awaiting`) and the League tab's second Matches tile (`matches_awaiting`). A trade is either awaiting *or* a mutual match, never both; it leaves this bucket the moment the match row is created.

**Mirrored trade** — Same player set viewed from both sides: A's "give X get Y" and B's "give Y get X" are mirrors.

**Trade block ("on the block")** — Players a manager has flagged in the Sleeper app's Trade Center as available to trade. Not in the documented v1 REST API, but publicly readable (no auth) via Sleeper's GraphQL `league_players` query — `settings.otb` is the flagging roster_id, `settings.otb_added_at` an epoch-ms timestamp (FB-147). FTF imports it per league during `session_init` (flag `sleeper.trade_block`, `backend/trade_block_service.py`), stores a replace-on-sync snapshot in the `trade_block` table, and surfaces an "ON THE BLOCK" badge on trade-card player rows via the additive `on_block` card field. Sleeper leaves stale `otb` flags in place after a player moves, so FTF keeps a flag only when the flagging roster still owns the player.

**Block boost** — FB-147 engine hook (`trade.block_boost`): the trade-engine consumer of the trade-block signal. A **SOFT, acquire-side** boost — every v2-orchestrated card whose acquire side holds ≥1 player the *counterparty* flagged on the block gets a flat bounded composite bump (`1 + block_boost_weight`, default +15%), applied in `_generate_trades_v2` after all gates so it only reorders acceptable trades and never rescues a gated one (same discipline as **Positional need fit**). Give-side and the user's own flagged players are out of scope (operator-approved acquire-side only). The signal loads once per generation via `trade_service._load_on_block_by_uid`, grouping `database.load_trade_block` rows by the flagging owner. Distinct from the display-only `on_block` per-player badge, which is what a boosted card reuses for client-side inspectability.

**Sleeper user / league ID** — Public Sleeper IDs. We never create accounts; the Sleeper username is the identity.

**Decision type** — `swipe_decisions.decision_type`: `'rank'` (3-player) or `'trade'` (trade card).

**Ranking method** — `users.ranking_method`: `null`, `'trio'`, `'manual'`, `'tiers'`, `'anchor'` (2026-07-10, the Pick Anchor wizard), or `'quickset'` (2026-07-12, #119 — the guided tier quick-set walk). How the user is building their rankings. On mobile this doubles as the **rank-home preference**: the Rank tab's launch destination (`useSession.rankingMethodPref`, device-local) — null shows the Build-your-board chooser (`RankHomeScreen`); the Settings "We steer ↔ You steer" slider changes it.

**Scoring format** — Stored as `'1qb_ppr'` or `'sf_tep'`. Affects which seed values are loaded and which `member_rankings` rows are visible. Per-league via `leagues.default_scoring`.

**Unlocked formats** — `users.unlocked_formats` JSON list — formats in which the user has met the gating threshold and unlocked Trade Finder.

**Tier overrides** — `users.tier_overrides`: per-format `{player_id: elo}` map of manual overrides applied on top of the trio-derived Elo.

**Contrarian** — A user's ranking that materially disagrees with the league or community consensus. Surfaced via `/api/trends/contrarian` and `/api/league/contrarian`.

**Tradeability** — 0–1 score on a player the user OWNS in the selected league: how easy/profitable it is to trade them away, derived from the Trends "easiest sells" gap (`user_elo − community_mean_elo`) scaled as `clamp01(0.5 + gap/800)` — gap ±400 Elo saturates, gap 0 (e.g. a seed-only never-really-ranked player) is a neutral half bar. Computed by `trends_service.compute_tile_trade_scores`, serialized on `GET /api/rankings`, rendered as the `TRADE` meter on Tiers tiles (TestFlight #71).

**Acquirability** — 0–1 twin of tradeability for a player a LEAGUEMATE owns: how easy they are to buy, from the Trends "easiest buys" gap (`user_elo − owner_elo`, falling back to the community mean when the owner hasn't published rankings), same scaling. Free agents get no score (not acquirable via trade). Rendered as the `GET` meter on Tiers tiles.

**Wrapped** — Year-end recap powered by `wrapped_events`. Event types: `swipe`, `trade_match`, `trade_accepted`, `trade_declined`, `tier_save`, `ranking_reorder`, `league_sync`.

**user_events vs wrapped_events** — `user_events` is the structured product analytics log (one row per meaningful action with denormalized hot-read columns mirrored on `users`). `wrapped_events` is the silent stream specifically for the year-end recap. Both are append-only.

**OG image** — Open Graph share image (1200×630 PNG) generated server-side by `og_image.py` for tier and trade share links.

**Notification kind** — Granular type recorded in `notification_events_log.kind` (e.g. `new_match`, `winback_dormant`). Maps to a user-facing **bucket** (`trade_matches` / `weekly_digest` / `reengagement`) via `get_pref_bucket()` in the push dispatcher; the bucket controls the user's toggle in `notification_prefs`.

**Quiet hours** — When `notification_prefs.quiet_hours_enabled = 1`, pushes that fire during the user's local night land in `notification_queue` instead. The hourly cron tick collapses each user's queued rows into one summary push at their local 8am and clears the queue.

**Cron tick** — External scheduler (Render cron) hits `/api/cron/realtime-tick`, `/api/cron/hourly-tick`, or `/api/cron/daily-tick`. Real-time drains queued pushes ready to deliver; hourly handles bundle drain + 8am summaries; daily runs digests and re-engagement scans.

**Dedup key** — Identifier used to avoid sending the same notification twice (e.g. `match_id` for `new_match`, week-stamp for digests). Stored in `notification_events_log.dedup_key`.

**Streak** — Current user's daily-activity streak (consecutive days with a meaningful event). Surfaced via `/api/me/streak`.

**Leaderboard** — League + Universal leaderboards rendered inside the League tab. Backed by `/api/leaderboard`.

**Untouchable** — A player the owner has tagged "never offer from my roster" (feedback #95). Stored in `asset_preferences` with `list_type='untouchable'`, per (user, league); flag `trade.preference_lists`. Hard give-side filter in trade generation (all paths, incl. likes-you injections). The owner can still *receive* anyone. Marked via long-press on a YOU SEND player in the mobile Matches tab (`/api/league/asset-prefs`). Distinct from a **skip** (`user_player_skips` — "I don't know this player", which removes a player from ranking trios/suggestions, per user+format).

**Target** — The acquire-side counterpart of an untouchable: `asset_preferences.list_type='target'` biases trade suggestions toward cards that bring the player back (capped composite reward, same flag).

**Account** — The durable identity anchor introduced by the account-auth plan P2 (`accounts` table, `backend/accounts.py`) and made PRIMARY by P2.6 (#116): a user can exist as an account alone, with league platforms attached as linked sources. Created the first time a user signs in with Apple/Google; binds (stickily, at most once) to a `sleeper_user_id`. Survives device loss: signing in with the same provider on a new phone restores the bound session. Deleted — along with all user-keyed data — by `DELETE /api/account` (App Store 5.1.1(v)).

**Working key** — The opaque `user_id` string every user-keyed table and service actually keys on (`users.sleeper_user_id` PK by historical naming). For Sleeper-first users it's the real Sleeper id; for account-only users (P2.6) it's the synthetic `acct_<account_id>`; demo sessions use `demo_user_*`. The engine never assumes the key is a real Sleeper id — the few flows that do (Sleeper league fetch, JWT claim match) are guarded against `acct_*` keys.

**Account-only user** — An Apple/Google-authed account with no linked Sleeper source (P2.6, feedback #116). Works under the `acct_<account_id>` working key with a real *empty* league (`no_league` / "No league linked"): rank/trios/tiers/anchors fully functional and persisted; trades/matches/league features structurally empty until a source is linked. Born verified (`verified_via='apple'|'google'`), which arms the P1/P2.5 gates against anyone else naming that key. Links a Sleeper username via `POST /api/account/link-sleeper` (Settings → Account), which merges the board per the P2.6 merge matrix and re-keys the session to the Sleeper id.

**Linked source** — A league-platform attachment on an account (P2.6): today the sticky-bound Sleeper username (`accounts.sleeper_user_id`); linked ESPN leagues will sit alongside it in the Settings → Account list. Distinct from a **linked identity** (the sign-in credential below).

**Linked identity** — One provider credential attached to an account (`linked_identities` row): `(provider, provider_subject)` where the subject is the provider's stable `sub` claim from a verified identity token — never the email (Apple only reveals email on first authorization; only a SHA-256 `email_hash` is stored). A verified linked identity marks the session `verified_via='apple'|'google'`, the P2 sibling of P1's `verified_via='sleeper'` Sleeper-JWT proof.

**Verified session** — A session that has *proven control* of its `user_id` (account-auth plan P1), as opposed to the default username-only session (Sleeper usernames are public, so an unverified session proves nothing). P1 proof = the SleeperConnect WebView captures the user's Sleeper JWT, `POST /api/sleeper/link` checks its `user_id` claim against the session **and** exercises the token once against Sleeper's authenticated GraphQL API (`sleeper_write.verify_token_live`) — Sleeper itself is the signature oracle, since the JWT is decoded without local signature verification. On proof: `sess["verified"]=True` + persisted `users.verified_at`/`verified_via`. Once any controller verifies a `user_id`, unverified sessions for it lose write access **and board-content read access** immediately (**first-verified-controller-wins**; reads = the P2.5 read gate, which has no grace but never denies users with no verified controller — see api-reference.md §"The read gate"); other unverified writes are grace-mode allowed-but-logged until `auth.enforce_verified_writes` flips (P3). Verified-only surfaces: `POST /api/trades/propose`, `POST /api/account/reset-rankings` (the squatter "reset my rankings" remedy).

**ESPN league (read-only import)** — A league imported from ESPN Fantasy via `POST /api/espn/link` (flag `espn.link`, [plan](plans/espn-league-linking-plan-2026-07-11.md)). Stored in `leagues` with `platform='espn'` (the PK holds the numeric ESPN league id); rendered with a small "ESPN" text badge in league lists. Read-only: rankings/tiers/trios fully work; trade features arrive with plan Phase 2; no ESPN write path ever. No draft-pick data exists on ESPN, so pick-inclusive features silently stay players-only for these leagues.

**MFL league (read-only import)** — A MyFantasyLeague league imported via `POST /api/mfl/link` (flag `mfl.link`, [plan](plans/multi-platform-linking-plan-2026-07-17.md)). Public leagues need **no auth**; crosswalks to Sleeper ids via `mfl_id` (DP's primary key, ~100% coverage). `platform='mfl'`, "MFL" badge. Its per-league **`wwwNN` host** must be resolved (from a pasted URL or via `api.myfantasyleague.com/{year}/home/{id}`'s 302). Unlike ESPN, MFL exposes **future draft picks** — stored raw (`leagues.platform_future_picks`) for the pick-inclusive follow-up, not yet engine-wired.

**Fleaflicker league (read-only import)** — A Fleaflicker league imported via `POST /api/fleaflicker/link` (flag `fleaflicker.link`). **Zero-auth** public JSON API; crosswalks to Sleeper ids via **`sportradar_id`** (from roster `externalIds`; DP's `fleaflicker_id` column is a decoy). `platform='fleaflicker'`, "FLEA" badge. Discoverable by account email (`/api/fleaflicker/discover`) — no credential held.

**Player-ID crosswalk** — The per-platform `{external_id → sleeper_id}` maps built from DynastyProcess `db_playerids.csv` (`espn_service.get_crosswalk`, 24h in-memory TTL, bundled-snapshot fallback), with a shared position-strict name+position fallback tier (#127). One fetch serves every linked platform: `by_espn_id` (ESPN), `by_mfl_sleeper` (MFL — DP's primary key, ~100%), `by_sportradar_id` (Fleaflicker, ~99.7%), `by_yahoo_id` (future). ESPN measured 99.2% on the value-relevant pool. `map_generic_rosters` is the one code path MFL/Fleaflicker use. Everything downstream stores only Sleeper ids; a roster player who misses both id and name tiers is **skipped and reported** in the link response's `report.unmatched` — never invented as a placeholder.

**Synthetic member id** — The `league_members.user_id` written for imported-league counterparties who aren't FTF users: `espn:{SWID}` (fallback `espn:{league_id}.t{team_id}`), `mfl:{league_id}.f{franchise_id}`, `flea:{league_id}.t{team_id}`. Deterministic across re-syncs; must never reach push/notification paths. The linking user's own team carries their real FTF `user_id` (bound via `leagues.espn_my_team_id` / `leagues.platform_my_team`).

**Power rankings** — Feedback #142/#144 (`GET /api/league/power-rankings`, math in `backend/power_rankings.py`, mobile `LeagueSummaryScreen` — League tab → "League rankings"): every team in a league ranked by **summed roster value**, deterministic order (total value desc, user_id tiebreak). Two live bases: `consensus` (universal-pool values — the `/api/trade/values` numbers) and `personal` (the caller's own board, consensus fallback for players they haven't ranked — so the same league ranks differently on different users' screens). A third basis, `redraft`, is reserved but returns 501 `not_available` (FTF's DynastyProcess value source is dynasty-only); the UI shows it as a disabled "(soon)" chip. Out-of-pool players (K/DEF, deep stashes) contribute 0 — a 1500-Elo default would hand every deep bench ~1000 phantom points. Tapping a team opens its roster grouped by position, value-sorted within each group (#144).

**Free agent finder** — Feedback #143 (`GET /api/league/free-agents`, logic in `backend/free_agent_service.py`, mobile `FreeAgentsScreen` — League tab → "Free agents"): the best available (unrostered) players in a league, ranked by the **caller's board value** — personal Elo where the user has one, consensus seed per unranked player — over the active format's universal pool minus every league member's roster (Sleeper, ESPN-imported and demo leagues alike). Each surfaced FA may carry a **drop suggestion**: the caller's lowest-valued *same-position* rostered player whose value is strictly below the FA's, shown with the add/drop value delta ("Drop: X (+312)"). PICK pseudo-players are never free agents; `pos_rank` ("FA RB3") is computed across all FAs so it's stable under the QB/RB/WR/TE filter pills.

**Entitlement** — A row in `entitlements` granting a user paid access, resolved read-time by `backend/entitlements.py`. Two values today: `pro` (the Pro gate list) and `ad_free` (ads-only relief; implied by `pro`). Written only by the billing webhook projector, promo reward granting, or the operator manual-grant routes — never from client receipts. `expires_at NULL` = perpetual (Founder, manual perpetual grants). Grants attach to the working key *and* the bound `account_id`, so they survive Sleeper re-links.

**Observe mode (entitlements)** — The rollout state where `monetize.entitlements` is on but `monetize.paywall` is off: every `check_pro` call logs `ENTITLE-OBSERVE user=… route=… would_block=…` and allows the request. The monetization sibling of the auth AUTH-GRACE rollout — produces measured would-block counts before any user is ever gated (foundation §2.4).

**Manual grant** — An operator-issued entitlement (`source='manual_grant'`, `granted_by='operator'`) via the `X-Cron-Secret`-guarded `/api/admin/entitlements/*` routes: comp a tester, a founder, or support-fix an account. Never touched by billing traffic (a refund can't claw back a comp); revocation flips status, never deletes.

**Subscription ledger** — The append-only `subscription_events` table: every RevenueCat/Stripe webhook stored verbatim (idempotent on provider `event_id`) before the projector applies it to `entitlements`. The audit source for subscription tracking; the RevenueCat dashboard is only a convenience view.

**Rank set** — A versioned, timestamped, published ranking list (`rank_sets` + `rank_set_entries`) owned by a contributor (user or publisher). Declares a `set_type` (`dynasty`/`rookie` at launch; `redraft`/`bestball` behind `ranks.set_types_extended`) and a scoring format. Adopting one (`seed`/`replace`/`track`, per-league) re-seeds the adopter's Elo board — the marketplace's core mechanic (rankings-marketplace plan).
