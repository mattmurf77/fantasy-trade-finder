# Silent bug sweep — 2026-05-22

Read-only review of cross-cutting silent-failure modes across `backend/`,
`mobile/src/`, and `web/js/app.js`. Excludes territory owned by the
backend, render, and api-layer reviews.

## Latent / risky patterns

- **`mobile/src/utils/deepLinks.ts:50-58`** — cold-start tap on a `/u/<username>` deep link arrives before `navigationRef.isReady()`. The `_navReady()` check returns false and the navigate is silently lost. (The react-navigation `linking` config in `RootNav.tsx:119-130` handles the cold-start path, so this is double-handled, but if anyone removes the listener thinking the linking config covers it, the warm-start path breaks silently.) Worth a comment.
- **`mobile/src/state/useNotifications.ts:32-48`** — `add()` doesn't dedupe by id. A foreground-received-then-tapped notification produces two events; both increment the badge. The cap to 50 papers over it.
- **`mobile/src/api/auth.ts:130-137`** (`initLeagueSession`) — `opponent_rosters` is filtered to drop any roster with zero players (line 137). For a brand-new league pre-draft, every roster has zero players and the trade engine sees no opponents at all. The user sees an empty Trades tab with no explanation. Defensive but silent.
- **`mobile/src/state/useSession.ts:105-110`** — bootstrap `JSON.parse` for `SU_KEY` / `SL_KEY` succeeds for any valid JSON. A corrupted blob (e.g. `{}` or `[]`) yields a typed object that doesn't have `user_id` / `league_id`, and downstream code happily references undefined fields. A `typeof parsed === 'object' && 'user_id' in parsed` shape check would harden this.
- **`mobile/src/api/client.ts:171-178`** — on 401, `clearSessionToken()` is awaited but `useSession.hasToken` is not updated. Until the next `bootstrap()` or `setLeague(null)` call, components reading `hasToken` still think the user is signed in and may fire queries that immediately 401-loop. The `progressQuery` in `RootNav.tsx:75-83` is gated on `hasToken`, so this matters.
- **`backend/server.py:3306-3307`** — `record_event` failures inside the disposition path are logged at WARNING. Same in 5+ other places. A persistent DB write failure that drops every analytics event would log a wall of warnings but the user-facing path keeps succeeding; the metric just goes to zero. Worth at least one ERROR-level log per N consecutive failures.
- **`backend/server.py:2178`** — `tiers.values()` runs without checking that `tiers` is a dict. A client sending `{"position":"RB","tiers":[]}` hits an `AttributeError` caught by the outer `except Exception` at 2235 and returns a 500 with the raw error string. Should reject with 400 + a friendly message at the top of the route.
- **`backend/server.py:2762`** — `not decision` validation also passes empty string but rejects the explicit value `0`; harmless today since values are strings, but the asymmetry is a footgun if the contract widens.
- **`backend/database.py:632-695`** — six bare `except Exception:` blocks inside the schema migration helpers. Migration silently no-ops when something fails (e.g. a partial schema upgrade from a previous half-applied migration). Should at least log at WARNING with the original exception.
- **`mobile/src/api/trades.ts:50-53`** — mismatch-score scaling assumes a `/300` cap. If `trade_service.py` ever widens its composite cap (it's a single `min(mismatch, 300) / 300` in trade_service), the mobile bar visualisation drifts silently. Add a TODO or pull the cap from the API.
- **`mobile/src/api/trades.ts:117`** — `getRecentTrades` is exported but never called anywhere in `mobile/src`. Dead surface area; either delete or wire up.

## What I checked and found clean

- **`mobile/src/state/useFeedback.ts`** — the hydrate fix from the brief is in place (line 116-119). Sync path correctly distinguishes "synced" from "has server_id" and `retrySync` re-reads state in case of mid-loop mutation. Good.
- **`mobile/src/screens/TiersScreen.tsx` coord handling** — the PR #60 fix (measureInWindow for screen-Y) is correct end-to-end. `zoneAt`, `dropTargetAt`, `onDropAt` all consistently use absolute Y. The chip layout side also uses measureInWindow (line 875).
- **`mobile/src/state/useSession.ts:switchLeague`** — the atomic check-acquire pattern inside `set()` (line 157-164) correctly prevents the read-then-set race. Comment explicitly calls out the prior bug.
- **`backend/server.py:_sleeper_get`** — TLS fallback is real and logged; non-silent. HTTPError/URLError paths re-raise rather than masking. Good.
- **`backend/database.py:check_for_match` / `match_already_exists`** — set-based comparison is order-tolerant; both orderings are checked. Logic is correct. (Unused `give_str`/`receive_str` vars at 2708-2709 are dead code but harmless.)
- **`mobile/src/api/trades.ts:normalizeTradeMatch`** — translation of `accept`/`decline` ↔ `accepted`/`declined` is bidirectional and covers both vocabularies. Defensive `String()` coercion handles int/string match_id drift.
- **TanStack Query `dispMutation` rollback** (MatchesScreen.tsx:86-89) — captures `prev` in `onMutate`, restores on error. Pattern is correct (other than the cross-cache invalidation gap in bug #5).
- **`usePushPriming` interaction with `requestAndRegister`** — clear() in `finally` block, registeredRef guard prevents double-prompt across `enabled` flips.
- **`backend/server.py:disposition_trade_match` league_id resolution** — line 3401-3402 correctly uses `result.get("league_id")` (the match's own league) for the refresh, not the session league. The notif builder a few lines up does NOT do this (bug #2 caveat).
- **`backend/ranking_service.py:reload_config`** — correctly filters by `_DEFAULT_CFG` keys when updating `_cfg`. (Compare to trade_service which doesn't — bug #8.)

## Open questions for the user

1. **Bug #1 confirmation.** Is the "active" → "pending" mismatch a known intermediate state from a partial migration, or has this always been the case? If migration: are there rows in production with `status='pending'` already (from a prior code path), or is the entire `matches_pending` count surface currently stuck at 0? Choice of fix depends on which.
2. **Bug #2 scope.** The disposition notification builder uses the **active session league** to look up the match, but the route accepts cross-league dispositions. Is this a latent bug we should fix proactively (always use the match's own league_id), or has nobody actually triggered a cross-league disposition with a "name-containing" notification yet?
3. **Bug #4 fallback policy.** If `/api/feature-flags` fails at boot, do we want flags to default to `false` (current), or to their last-cached values (which would require local persistence)? The current behaviour means a backend outage hides every gated feature for the duration.
4. **Bug #6 push-kind drift.** Should the push-tap handler default to navigating somewhere sensible (e.g. Matches) when it sees an unknown kind, or stay silent and rely on the in-app feed? The latter is current behaviour and means new kinds appear invisible until both apps ship.
5. **Bug #7 validation strictness.** `/api/trades/swipe` accepting any non-empty string as decision is permissive — do we want to add an explicit `in ("like","pass")` check and return 400, or keep permissive for client-version compatibility?
