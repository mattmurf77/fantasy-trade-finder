## What the user reported

> "The app just loads too slowly when rendering an experience for the first time (the trios prompt, etc)."

## Why Trios actually feels slow (root cause)

The slowness is **not one bug**. It's a stack of four:

1. **No skeleton** — `RankScreen.tsx:368-371` replaces the entire card area with a bare `ActivityIndicator` until `/api/trio` resolves. Even a 700 ms warm response feels like "nothing then everything" (Mobile #M1).
2. **No prefetch on the menu→screen transition** — `RankMenu.go()` (`TabNav.tsx:159-167`) only dispatches navigation; the `/api/trio` fetch doesn't start until `RankScreen` mounts. The 250–400 ms modal-close animation is dead air (Mobile #M2).
3. **`['progress']` refetches every trio submit** — `RankScreen.submitMutation.onSuccess` invalidates `['progress']`; on already-unlocked users that's a wasted ~250 ms round-trip per swipe (API #A3 + Backend #B5).
4. **Backend re-does the unlock write txn on every poll** — even after the user is unlocked, `/api/rankings/progress` runs `mark_format_unlocked` + peer-push fanout per poll (Backend #B5).

Cold-dyno wake (30–60 s on Render free tier) still dominates first-launch — the $7/mo no-sleep dyno is the only complete fix for that, but it's the user's call. Everything below assumes warm dyno.

## P0 — Production bug found during review (ship immediately, no relation to Trios)

**Status string mismatch breaks 3 cron paths** (Silent #C-P0).
`backend/database.py:2768` writes `status="active"` on new `trade_matches`. Three readers filter on `status="pending"` (`database.py:2061, 4599, 4625`). Net effect:
- `match_expiring` push cron scans 0 matches forever
- `pending_review` Tue/Wed digest never fires
- `matches_pending` badge counter always 0

User-facing Matches inbox masks this with a read-side string normalization, so it looks like it works. But every cron job that targets pending matches is silently a no-op.

**Fix:** one-line — write `"pending"` (or normalize the readers to accept both with a tracked migration). ~5 lines, 1 PR, single-file change.
