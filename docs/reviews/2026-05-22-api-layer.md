## Duplicate / wasteful fetches

1. **`['rankings']` over-invalidation cascades to TiersScreen.**
   `ManualRanksScreen.tsx:105` invalidates the entire `['rankings']` family
   on every debounced reorder, which refetches `getRankings(null)` AND every
   `['rankings', position]` TiersScreen holds. A user touching both screens
   pays a 4-position refetch per 600ms debounce. Narrow to the affected
   position, or pass `refetchType:'active'`.

2. **`['progress']` keeps refetching after unlock.** `RootNav.tsx:75-82`
   gates new observers via `everUnlockedRef`, but
   `RankScreen.submitMutation.onSuccess` (`RankScreen.tsx:103`) invalidates
   `['progress']` on every trio submit — invalidation refetches an existing
   cache entry regardless of `enabled`. Post-unlock, every rank3 still hits
   `/api/rankings/progress`. The response already carries
   `interaction_count/threshold/percent`; use `setQueryData` instead of
   invalidate (the streak path at L117 already does this).

3. **TiersScreen position pill = network call per tap.** `TiersScreen.tsx:102-106`
   keys on `position` with no `placeholderData`. QB→RB→WR→TE = 4 round-trips
   even though `['rankings','all']` (from Overall/Manual) holds everything.
   Switch to the `'all'` key + `select:` to filter — see win #A5.

4. **`SettingsScreen` double-writes notif-prefs.** `SettingsScreen.tsx:53-67`
   keeps a `local` mirror AND calls `setQueryData(['notif-prefs'], next)` on
   success. Two writes per toggle; pick one as truth.

5. **`useFeedback.retrySync()` fires foreground without hydrate first.**
   `App.tsx:100-110` runs `retrySync` on every AppState 'active'. If the
   user hasn't opened FeedbackInboxScreen this session, `items` is `[]` and
   the retry early-returns — meaning unsynced notes from a *prior* session
   never replay. Call `useFeedback.getState().hydrate()` first (idempotent).

6. **`getNewPartners` re-fetches activity instead of sharing cache.**
   `mobile/src/api/league.ts:282-303` calls `getActivityFeed(leagueId, 50)`
   while LeagueScreen holds `getActivityFeed(leagueId, 10)` under
   `['league-activity',leagueId]`. Different limits → no cache reuse. Either
   bump LeagueScreen to 50 and share, or add a `/api/league/partners`.

7. **ProfileScreen 404s sit forever.** `ProfileScreen.tsx:50-56` skips retry
   on 404/400 but no `gcTime` cap — a missed profile stays in cache.

## Top 5 actionable wins (ROI ordered)

### #A1 — `/api/notifications/read` body shape mismatch

- **Symptom:** Any code path that wires up "mark a single notification read"
  silently no-ops; backend returns `{ok:true, updated:0}` because it never
  finds an `ids` key in the body.
- **Cause:** `mobile/src/api/notifications.ts:12-16` sends
  `{notification_id: <single>}`; backend at `server.py:4881-4895` reads
  `body.get("ids")` (a list).
- **Fix:** Change wrapper to `api.post('/api/notifications/read', { ids:
  [notificationId] })`. While there, mirror the same `{ids?:number[]}` shape
  on the wrapper type so future contracts stay tight.
- **Effort:** trivial. **Impact:** medium (latent — no caller today, but the
  bell-sheet's per-row "mark read" is on the roadmap). **Risk:** very low.

### #A2 — Invalidate ranking caches after writes that change ELOs

- **Symptom:** Submit a trio, save tiers, or reorder ranks; tab over to Tiers
  or Overall — old ELOs / order. Has to wait 30 s or navigate to refresh.
- **Cause:** `RankScreen.submitMutation.onSuccess` only invalidates
  `['progress']` + `['trio',position]`; `TiersScreen.saveMutation.onSuccess`
  invalidates `['tiers-status']` + `['progress']`. Neither touches the
  `['rankings', …]` family the other screens read.
- **Fix:** Add `queryClient.invalidateQueries({queryKey:['rankings']})` to
  both onSuccess handlers. To avoid stampeding (see #A3), set
  `refetchType:'active'` so only currently-mounted screens refetch.
- **Effort:** small. **Impact:** medium-high (this is the single biggest
  reason users see stale data on tab switches). **Risk:** low.

### #A3 — Stop firing `['progress']` invalidations once unlocked

- **Symptom:** Every Trios submit re-fetches `/api/rankings/progress` on
  Render's free tier — ≈200 ms warm, multi-second cold. Adds latency to the
  next trio fetch (next trio + progress are sequential on the deck-empty path).
- **Cause:** `RankScreen.tsx:103` invalidates `['progress']`. `RootNav.tsx:75`
  also observes that key. The `everUnlockedRef.current` short-circuit only
  prevents *new* observers from running; an invalidation against an existing
  cached entry refetches regardless.
- **Fix:** Make `RankScreen.submitMutation.onSuccess` do
  `queryClient.setQueryData(['progress'], (prev) => ({...prev, [position]:
  resp.interaction_count}))` instead of `invalidate`. Once unlocked, skip the
  refetch entirely. Already mirrored for `['streak']` at L117 — same pattern.
- **Effort:** small. **Impact:** medium (perceived snappiness on every trio
  submit). **Risk:** low — fallback `invalidate()` still works if you don't
  trust the response shape.

### #A4 — Add invalidation set to `useSession.switchLeague` /
       `useSession.connectLeague`

- **Symptom:** After switching leagues, MatchesScreen's optimistic cache
  shows last-league state for up to 15 s (the staleTime); Portfolio cache
  doesn't refresh on connect even though server state changed.
- **Cause:** `useSession.ts:146-178` mutates local state but has no handle
  on the QueryClient. The `[leagueId]` keys auto-refetch — but stable keys
  (`['matches','all']`, `['portfolio']`, `['liked-trades',leagueId]` after
  a connect) don't.
- **Fix:** Add a `queryClient` argument to `switchLeague` /
  `connectLeague` (or expose them via a hook that has the queryClient
  available) and `invalidateQueries({queryKey:['matches']})`,
  `invalidateQueries({queryKey:['portfolio']})`, `invalidateQueries({queryKey:
  ['awaiting-trades']})`. Alternatively, subscribe to a "league changed"
  zustand event from a `QueryInvalidator` component mounted under
  `<QueryClientProvider>`.
- **Effort:** small. **Impact:** medium. **Risk:** very low.

### #A5 — TiersScreen position pill should not refetch — share
       `['rankings','all']` with Overall/Manual

- **Symptom:** Tapping QB → RB → WR → TE in Tiers fires 4 network calls,
  each ≈150-400 ms warm. The data is already cached under
  `['rankings','all']` whenever Overall or Manual has been visited.
- **Cause:** `TiersScreen.tsx:102-106` keys on `position`. Different cache
  key from the all-list. No `placeholderData`.
- **Fix:** Switch to `useQuery({queryKey:['rankings','all'], queryFn: () =>
  getRankings(null), select: (data) => filterByPosition(data, position)})`.
  TanStack's `select` runs client-side per render, returns the same cached
  Rankings object filtered to one position. The `useEffect` at L193 already
  re-buckets on `[rankingsQuery.data, position]` so the select migration is
  transparent.
- **Effort:** small. **Impact:** medium (TF: under 1s page-changes vs 1-3s
  today). **Risk:** low — same data either way.

**Observations:**

- Only 7 of 25 queries use `placeholderData:(prev)=>prev` — every screen
  outside `LeagueScreen` and `MatchesScreen` will blank on re-entry. The
  perf-audit recommended adding it on LeagueScreen + Matches (done); the
  same fix is one-liner on TiersScreen (#5 above), TrendsScreen, Trades, and
  RankScreen-trio.
- Only ProfileScreen uses a custom `retry`. Most queries fall back to the
  default `retry:1`, which is fine for transient blips but ends up doubling
  the latency on a hard 5xx. Worth scoping a tighter retry to mutations
  that can be re-tried client-side (`reorderRankings`, `saveTiers`).
- `select` is not used anywhere. Several screens (#5 above) would benefit
  from sharing a cached payload across position filters.
- `refetchOnWindowFocus` is off globally, on only at the root `['progress']`
  observer. Reasonable for a mobile app; not worth turning on elsewhere
  without backing data.
- No query uses `refetchInterval` — the only polling is the imperative
  `setInterval` in TradesScreen's trade-status loop (already in the perf
  audit's deferred list with no-backoff flag at 1.5 s).

## Web vs mobile parity diffs

- **`/api/sleeper/players` heavy path.** Mobile uses `/warm`
  (`api/sleeper.ts:43-49`). Web (`app.js:661, 792, 1141, 2405`) still hits
  the 4.8MB endpoint every league flow. Mirror the warm path on web.
- **Trade-status poll cap.** Both poll at 1500 ms with no backoff
  (`TradesScreen.tsx:254`, `app.js:3072-3126`). Web caps total polls at 40;
  mobile caps only on consecutive failures (4). A stuck job on mobile polls
  forever until unmount — add `MAX_POLLS=40`.
- **Notifications polling.** Web polls every 30 s (`app.js:4398`). Mobile
  doesn't poll; `getNotifications` is never called. Push-denied users get
  nothing until restart. Wire `useQuery(['notifications',userId],
  {refetchInterval: 60_000, enabled: !!userId && !pushPermissionGranted})`.
- **Demo flow.** Web uses raw `fetch` (`app.js:1205`), bypassing the device-
  header injection that lives in `apiFetch`. `users.last_device_type` is
  always empty for web demo signups. Route through `apiFetch`.
- **Abort signals.** `apiRequest` accepts `opts.signal` (`client.ts:127`),
  but no wrapper threads through the `AbortSignal` TanStack injects into
  the queryFn context. A user changing screens mid-fetch can't cancel
  in-flight requests. Cheap win: `queryFn: ({signal}) => getX(args, {signal})`.
  Web client doesn't use AbortControllers either.
- **`sessionPing`** wrapper exists in mobile (`api/auth.ts:86-88`) but no
  caller. Looks like dead code; web uses it on boot for pre-flight liveness.
  Mobile relies on the lazy 401 handler. OK; flag for next reader.

## Error handling

`ApiError` (`client.ts:106-118`) carries `status`, `body`, `message`, but
callers mostly throw away the structured info:

- `MatchesScreen.tsx:88` — "Action failed — try again" for everything.
- `RankScreen.submitMutation` — **no `onError` at all** (`RankScreen.tsx:94-120`).
  Submit fails → trio rotates anyway, user has no indication.
- `TradesScreen.swipeMutation.onError` not implemented (`TradesScreen.tsx:291-301`).
  Swipe POST fails silently; deck advances regardless.
- `TiersScreen.tsx:142-143` — uses `err.message` (better) but no
  distinction between offline and server error.

Suggested baseline: a `toUserMessage(err: unknown)` helper in
`api/client.ts` returning "Lost connection" for network errors, "Session
expired" for 401 (and call `signOut()`), and the server message otherwise.
Wire into every `onError`.

## Optimistic mutations — rollback hygiene

- `MatchesScreen.dispMutation` (`MatchesScreen.tsx:73-85`) — snapshot `prev`,
  filter, restore on error. Correct ✓.
- `TiersScreen.dismissMutation` (`TiersScreen.tsx:177-190`) — **no `onError`
  rollback**. A failed dismiss leaves the chip gone until next screen
  remount. Add an `onMutate` snapshot of `buckets` + restore in `onError`.
- `SettingsScreen.mutation` (`SettingsScreen.tsx:59-63`) — restores `local`
  from `prefsQuery.data`. Two-quick-flip race: first POST fails after
  second `local` write → rollback discards both flips. Fix: snapshot
  `local` (not `prefsQuery.data`) inside `onMutate`.
- `TradesScreen.swipeMutation` — no optimistic update; deck advance is pure
  UI. Combined with missing onError, liked trades can silently fail to
  register on the server.
