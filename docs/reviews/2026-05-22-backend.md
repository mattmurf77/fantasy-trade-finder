# Backend review — 2026-05-22

Scope: Hot-path perf + silent bugs in `backend/server.py` (6697 lines),
`backend/database.py` (4828 lines), `backend/trade_service.py` (1362),
`backend/ranking_service.py` (859), `backend/trends_service.py` (421),
`backend/feature_flags.py` (223).

Focus on routes the mobile app cold-starts on:
`/api/session/init`, `/api/sleeper/players/warm`, `/api/trio`,
`/api/rankings`, `/api/trades/generate` + `/status`,
`/api/league/coverage`, `/api/league/members`.

## Top 5 findings (ROI ordered)

### #B1 Missing indexes on the swipe/trade decision hot path

- **Symptom**: `session_init` for a returning user latency is dominated by
  two queries that scan the entire `swipe_decisions` and
  `trade_decisions` tables for one user's rows. As tables grow (every
  ranking writes 3 rows per trio; every trade swipe writes N×M
  pairwise rows), cold load gets steadily slower for everyone.
  `check_for_match` (fires on every "like" swipe) and
  `load_trade_decisions` (session_init, 7-day window) both repeat the
  pattern.
- **Root cause** (file:line):
  - `backend/database.py:105` `swipe_decisions_table` declares **zero
    indexes** — no `user_id`, no `(user_id, scoring_format)`. The
    primary key is just `id`.
  - `backend/database.py:117` `trade_decisions_table` same — only
    `id`. `check_for_match` at `database.py:2672` does
    `SELECT * FROM trade_decisions WHERE user_id = ? AND league_id = ?
    AND decision = 'like'` per swipe, full-scan today.
  - `backend/database.py:144` `member_rankings_table` — no index on
    `(league_id, scoring_format)` or `(user_id, league_id)`. Every
    `/api/trades/generate` job loads via `load_member_rankings`
    (`database.py:2537`) with a full scan filtered by `league_id` +
    `scoring_format`.
  - `backend/database.py:361` `elo_history_table` — no index on
    `(user_id, scoring_format, snapshot_at)`. `/api/trends/risers-fallers`
    queries this on every Trends tab visit
    (`server.py:5543`); full scan today.
- **Proposed fix**: Add five composite indexes (via
  `_migrate_db` so existing DBs upgrade): `ix_swipe_user_format` on
  `(user_id, scoring_format)`, `ix_trade_dec_user_league` on
  `(user_id, league_id)`, `ix_trade_dec_user_dec` on
  `(user_id, decision)`, `ix_member_rankings_league_fmt` on
  `(league_id, scoring_format, user_id)`, and
  `ix_elo_hist_user_fmt_at` on
  `(user_id, scoring_format, snapshot_at)`. SQLite `CREATE INDEX IF NOT
  EXISTS` is safe; same DDL works on Postgres.
- **Effort/Impact/Risk**: small / large / small. Indexes alone shave
  most of the per-DB-row scan cost. Risk: extra write amplification on
  swipe inserts is negligible for these tables.

### #B2 `/api/session/init` does serialized blocking work that could parallelize or defer

- **Symptom**: Mobile cold-start says "trios prompt loads slowly." The
  request that gates the first paint of *every* tab is `session_init`,
  which on a returning user with non-trivial history performs (in order,
  all on the request thread):
  1. `_ensure_universal_pools()` — building two ranking pools
     (`server.py:4402`).
  2. For each scoring format (×2): `load_swipe_decisions` (full-table
     scan, see #B1), `RankingService.replay_from_db`, then a separate
     `load_tier_overrides` JSON parse. `server.py:4493–4546`.
  3. `load_trade_decisions(since_days=7)` (full-table scan, see #B1)
     to seed the past-decision dedup set. `server.py:4561`.
  4. `upsert_user` + `upsert_league` + `record_event`
     (`server.py:4628–4658`).
  5. `_fetch_sleeper_league_meta` — **outbound HTTP to Sleeper, every
     init, blocking** (`server.py:4740`). Comment explicitly says
     "Re-detect scoring format on EVERY session_init … cost is one
     cached Sleeper API call per init" — but `_sleeper_get` cache is
     not visible here and the comment lies if the cache misses.
  6. `load_league_member_unlock_states` + per-peer
     `_send_typed_push` on new-user branch (`server.py:4695–4716`).
  7. `upsert_league_members` (N × SELECT + N × INSERT/UPDATE inside one
     transaction — `database.py:1930`).
  8. Pre-generate trade-card kickoff (`server.py:4795`) — daemon
     thread, so non-blocking; good.
- **Root cause**: Steps 2/3/5/7 are blocking and dominate latency for
  any user with stored history or a real Sleeper league. Each format's
  replay+overrides happens sequentially when they're independent.
- **Proposed fix** (separable, each small):
  - Run the two per-format `RankingService` rebuilds in parallel via a
    `ThreadPoolExecutor(max_workers=2)`; both are CPU-light + DB-IO.
  - Move `_fetch_sleeper_league_meta` + `set_league_scoring`
    detection to the background daemon thread that already kicks off
    pre-gen — first-init latency goes down, retry semantics preserved
    by next init.
  - Replace `upsert_league_members`'s row-by-row loop with one bulk
    `INSERT … ON CONFLICT DO UPDATE` (SQLite ≥3.24 + Postgres both
    support this). Cuts ~24 round-trips for a 12-team league to 1.
  - Move `load_league_member_unlock_states` + peer pushes out of the
    request — append to a `notification_queue` row and let the existing
    cron tick drain it. Push is best-effort already.
- **Effort/Impact/Risk**: medium / large / medium. Each piece is small
  but they add up; the parallel `RankingService` rebuild needs a brief
  check that `_get_universal_pool` is thread-safe (it caches into
  `g_universal_by_format` — guard with a lock).

### #B3 `disposition_trade_match` notification builder uses the wrong key — silent fail (already filed)

- **Symptom**: After a mutual trade match accepts/declines, both users'
  "trade accepted/declined" in-app notifications render with empty
  player names ("…: " with nothing on either side of "for").
- **Root cause** (`server.py:3356`):
  ```py
  _raw_ms = load_matches(user_id=g_user_id, league_id=...)
  _this_m = next((m for m in _raw_ms if m["id"] == match_id), None)
  ```
  but `load_matches` returns dicts keyed `"match_id"` (`database.py:2895`),
  never `"id"`. The `next()` returns `None`, the subsequent comprehensions
  swallow it silently:
  ```py
  _gv_names = ([_pd[pid].name for pid in _this_m["my_give"] if pid in _pd]
               if _this_m else [])
  ```
  so both player-name lists stay empty and the formatted body collapses
  to `"✅ Joe accepted your trade in League: "`. This is a representative
  instance of an anti-pattern that appears repeatedly: lookup-with-default
  + downstream conditional that hides a contract drift.
- **Proposed fix**: `m["match_id"] == match_id`. Better: extract a
  `_match_by_id(matches, match_id)` helper that asserts when the dict
  shape it expects is missing, so the next field-rename is caught loudly.
- **Effort/Impact/Risk**: small / medium / small. Already queued for
  follow-up per the review brief; calling out as a representative
  pattern, not a new bug.

### #B4 `/api/league/members` and `/api/league/member-unlock-states` redundantly call the same heavy loader

- **Symptom**: Web/mobile League screen renders both
  `/api/league/members` and `/api/league/member-unlock-states` on
  first paint. Each calls
  `load_league_member_unlock_states(league_id, exclude_user_id=g_user_id)`
  (`server.py:3724` and `server.py:3774`), which runs two SELECTs
  (`league_members` + `users.in_(member_ids)`) plus JSON parsing.
  Doubled work, doubled DB chatter, on every screen view.
- **Root cause**: `/api/league/members` is implemented as a
  field-projection over the same data — the comment at
  `server.py:3770–3777` calls this out explicitly ("Reuse existing
  per-member loader … keeps a single source of truth"). Convenient
  but wasteful when the client fetches both endpoints.
- **Proposed fix**: One of:
  - Memoize `load_league_member_unlock_states(league_id, exclude)` at
    module scope with a tiny TTL (e.g. 30s) and cache key
    `(league_id, exclude_user_id)`. Same pattern as
    `_LEADERBOARD_CACHE` at `server.py:1384`. Both endpoints share
    the cached result.
  - Or collapse to a single endpoint that returns both shapes and
    deprecate one of them in the next client release.
- **Effort/Impact/Risk**: small / medium / small. The cache approach
  is a 10-line addition; matches an existing pattern in the file.

### #B5 `mark_format_unlocked` + push fanout runs every `/api/rankings/progress` poll

- **Symptom**: The mobile app polls `/api/rankings/progress` on the
  Rankings tab to update the "Unlock Trade Finder" CTA. For any
  already-unlocked user, every poll still:
  - takes the `engine.begin()` write transaction in
    `mark_format_unlocked` (`database.py:1641`) — `SELECT FOR UPDATE`
    on Postgres, write lock on SQLite — even when the format is
    already in the list.
  - re-loads `load_league_member_unlock_states` and iterates peers to
    fire `_send_typed_push` calls (each guarded by `_freq_cap_blocks`
    and dedup, so the network push is suppressed, but the DB read +
    insert into `notification_events_log` still happens).
- **Root cause** (`server.py:1709–1753`): the `if unlocked:` branch is
  taken unconditionally on every poll once the user has unlocked, then
  fires the peer-push fanout when `was_first` is True. The check that
  `was_first` was already false is inside `mark_format_unlocked` so the
  outer flow still pays for the write txn and the peer load.
- **Proposed fix**: short-circuit at the top of the `if unlocked`
  branch when `fmt in unlocked_formats_list` (the value already
  loaded at `server.py:1694`) — skip `mark_format_unlocked` and the
  peer-push fanout entirely. Only run the write path on the transition.
- **Effort/Impact/Risk**: small / medium / small. Trio of one-line
  guards; preserves first-time-unlock semantics.

## Lower-priority findings

- **L1** `_fetch_sleeper_league_meta` runs on every `/api/session/init`
  even when the league's scoring format is already on file
  (`server.py:4730–4747`). Comment claims re-detection is "self-healing"
  but cost is paid on every login. Gate behind a 24h timestamp on the
  league row.
- **L2** `upsert_league_members` (`database.py:1930`) runs N selects +
  N upserts inside one transaction. Use a single
  `INSERT … ON CONFLICT DO UPDATE` (see #B2).
- **L3** `_ensure_sleeper_cache_populated` (`server.py:4261`) holds
  `_player_sync_lock` only around the `sync_players` call but the
  Sleeper HTTP fetch above it is not guarded — two cold clients arriving
  simultaneously fetch ~5MB from Sleeper twice. Hoist the lock above
  the `urlopen`.
- **L4** `load_trade_decisions(since_days=7)` (`database.py:1895`)
  filters by ISO string compare on `created_at`. Works because the
  format is sortable, but a real `DateTime` column would let Postgres
  use a proper index when added. Also `since_days` is computed at
  request time but the SQL builds a string — won't use partial index.
- **L5** `_run_trade_job` resolves the session by token AFTER spawning
  the thread; if `session_init`'s caller's token was rotated between
  enqueue and worker pickup (e.g. concurrent login), the job just
  "expires" silently with `log.exception`. Low-probability but no
  retry path.
- **L6** `/api/trades/matches/all` (`server.py:3025` — quick scan) and
  `load_matches(league_id=None)` (`database.py:2794`) is the right
  shape, but the `league_members_table.c.league_id.in_(...)` query at
  `database.py:2845` is unbounded by user — a user in 30 leagues
  causes one giant IN list and pulls every member of every league.
  Acceptable for now (small leagues) but worth a `(league_id, user_id)`
  projection or a `join` to scope to needed (league, user) pairs only.
- **L7** `_LEADERBOARD_CACHE` (`server.py:1384`) has no eviction.
  Unbounded growth keyed on `(scope, league_id, metric, window)`.
  Practical bound is small but worth an `OrderedDict` + max size.
- **L8** `get_trio` falls back to `service.get_next_trio` with a broad
  `except Exception as e: return jsonify({"error": str(e)}), 400`
  (`server.py:1374`). Returns 400 with a raw exception string — fine
  for an unrecoverable user-side mistake but masks server-side bugs
  (e.g. pool corruption) as client errors.
- **L9** `_ensure_universal_pools` (`server.py:742`) is correctly
  idempotent — early-returns when both formats are built (verified
  at `server.py:750`). However it is NOT thread-safe: two cold
  `session_init` calls racing into the loop at `server.py:758` could
  both `build_universal_pool` for the same format and double-write
  `g_universal_by_format[fmt]`. Wrap the function body in a module-
  level lock. Latency consequence is duplicate ~2s pool builds on
  the first two cold concurrent users.
- **L10** `swipe_trade` writes BOTH `save_trade_decision` and
  `save_trade_swipes` then conditionally writes a `trade_match` row
  inside one Python `try`, with no transaction wrapping them as a
  unit. A failure between the two leaves the DB inconsistent
  (decision recorded, swipes missing — or vice versa). Wrap the pair
  in one `engine.begin()`.
- **L11** `record_event` is called from every swipe/disposition path
  inside a `try/except` that only logs. If the events DB is down the
  user request still succeeds — good — but the warning floods logs.
  No backoff or circuit.
- **L12** `_freq_cap_blocks` (`server.py:5050`) presumably queries
  `notification_events_log` per push — and we fan push out per peer
  inside `/api/rankings/progress` (#B5) and per peer at session_init
  (`server.py:4695`). That's O(peers) DB reads on a hot route.
- **L13** `/api/trio` carries an `is_enabled("swipe.qc_compliments")`
  branch that calls `find_qc_trio` which calls `service._pool(...)`
  and then a list-comp over `skipped_ids` — runs on every trio
  request even between QC trios. Hoist the `_pool` slice into the
  service to avoid the per-call alloc.

## Silent-bug catalogue

Every silently-swallowed exception or quietly-skipped path with file:line.
The risk pattern is identical across all of these: a single API call
fails, a log line lands, the user sees stale or wrong UI. Worth a
follow-up pass to upgrade selected ones to errors that surface.

`backend/server.py`:

- `3795` `league/members` — 500.
- `4313` `_ensure_sleeper_cache_populated` → `_player_sync_lock` block —

- `4567` `load_trade_decisions` past-decision seed — `log.warning`.

- `4660, 4688, 4716, 4751, 4753, 4777` — six more `log.warning` blocks

- `4817` pre-gen kickoff — `log.warning`. OK.

`backend/database.py`:

- `1632, 1665, 1696, 2215, 2687, 2865, 2993` (estimated) —
  `except (json.JSONDecodeError, TypeError):` on every JSON-blob
  column read. Returns `[]` or `{}`. If a write ever corrupts a blob,
  the user just sees missing data and there's no logged signal at all.
  At minimum add a `log.warning` with the user_id and blob key.
- `1756` `wrapped_collector.record_event` from `upsert_league` — bare

## What I checked and found clean

- `trade_matches_table` has the correct composite indexes
  (`database.py:183–192`) — both `(user_a_id, league_id)` and
  `(user_b_id, league_id)` are present, so `/api/trades/matches/all`'s
  `OR` filter at `database.py:2818` is index-served on each side.
- `user_events_table` has the right indexes
  (`(user_id, occurred_at)` and `(event_type, occurred_at)` at
  `database.py:439–440`). Leaderboard reads are covered.
- `notification_events_log` has `(user_id, kind, sent_at)`
  (`database.py:491`). `_freq_cap_blocks` is index-served.
- `app_feedback_table` has both indexes (`database.py:328–329`).
- `_LEADERBOARD_CACHE` is a healthy pattern (TTL'd module-level cache
  for a hot read) and is used correctly for the per-scope key.
- `_trade_jobs_lock` is held for the right boundaries — entry/exit
  only, never during the worker call (`server.py:2679–2709`).
- `_kickoff_trade_job` correctly marks pinned-give jobs as
  non-cacheable (`server.py:1229`).
- `_run_trade_job` catches and surfaces all exceptions on the job
  status, so the polling client sees an explicit error
  (`server.py:1157`).
- `/api/trades/status` (`server.py:2712`) does user_id verification
  to prevent cross-user job-id leaks. Good defensive check.
- `/api/sleeper/players/warm` (`server.py:4338`) does the right thing

  this instead of the 5MB `/api/sleeper/players`. (One follow-up:
  see L3 re: the lock placement.)

## Open questions for the user

1. **Postgres or SQLite in prod?** Index choices and the `ON
   CONFLICT DO UPDATE` rewrite (#B2) work on both, but if you're on
   Postgres-only in prod, we can use partial indexes and `JSONB` on
   `tier_overrides`/`unlocked_formats` and remove a lot of JSON-blob
   parsing.
2. **Are mobile cold-start traces available?** I'd like to validate the
   #B1/#B2 ranking — happy to draft the SQL `EXPLAIN` snippets to run
   against a copy of prod to confirm that swipe_decisions is full-scanned
   in practice (vs. the dataset being small enough it doesn't matter
   yet).
3. **Push notification deliverability** — #B5 + L12 suggest the push
   fanout is a hot path. Is `_send_typed_push` actually doing
   networked sends synchronously per recipient, or queueing? (Didn't
   pull the body — flag-driven.)
4. **Is `load_matches`'s `"match_id"` vs `"id"` consistency the only
   key-shape drift?** Worth a small grep for similar "field renamed at
   one layer" cases — silent-bug pattern #B3 likely has siblings.

## Notes / methodology

- Read counts: ~1,500 lines of `server.py` (cold-start routes +
  surrounding helpers + disposition path), `database.py` schemas plus
  all loaders called from cold-start routes, plus the relevant chunks
  of the trade job worker. Did not exhaustively read
  `trends_service.py`, `ranking_service.py`, or
  `trade_service.py` bodies — focused on the route layer entry points
  per the brief's "hot routes only" guidance. If the user wants a
  deeper pass on the trade-math internals (e.g.
  `TradeService.generate_trades` complexity profile), that's a
  worthwhile follow-up review on its own.
- This report intentionally biases toward DB-level wins (indexes,
  query consolidation, request-time blocking work moved to background)
  because they have the highest ROI for cold-start "trios prompt loads
  slowly." A second pass focused on `RankingService` and
  `TradeService` algorithmic complexity would round out the picture.
