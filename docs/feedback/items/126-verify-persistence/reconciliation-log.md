## Blocking rebuttal (precedes the four questions — it moots two of them)

**Plan §2a/§2b: server-side `_maybe_autoverify(sess)` from the stored credential.** **REBUTTED — design replaced** (see [prd.md §0](prd.md)).

The plan's safety argument rests on §2b: *"A genuine squatter has no stored credential under the victim's user_id (storing one requires the claim-match + oracle at link time), so auto-verify gives them nothing."* The parenthetical answers the wrong question — whether a squatter can *store* a poisoned credential (they can't) — not whether a credential *already exists* at the lookup key. It does, for exactly the population the fix targets: `get_sleeper_credential(user_id)` is keyed by user_id alone ([database.py:5776](../../../../backend/database.py)), the squatter's session carries the victim's user_id by definition, and session mint needs only a public username ([server.py:8892-8899](../../../../backend/server.py); `session_init` likewise). So the helper's whole proof chain — victim's stored token, claim trivially matching the victim's user_id, oracle passing because the token is genuinely live — verifies **any session that names the username**. Steps 1-5 of plan §2a contain no fact about the session *holder*.

Blast radius if shipped: reopens the P1 write gate and P2.5 read gate to username-only attackers for every verified user, and — new, worse than pre-P1 — hands them `POST /api/trades/propose` (verified session + the route pulls the same stored credential, [server.py:5134-5153](../../../../backend/server.py)): real trades from the victim's Sleeper account. This directly violates the operator directive ("don't weaken the gate") and falsifies plan risk #4's reassurance that the change "widens *when* verification is granted, not *what* grants it."

**Replacement (PRD R-0..R-2):** persistence moves to the client — the mobile app keeps the captured JWT in the device Keychain (expo-secure-store, already a dependency) and silently **replays the existing hard-verified route** `POST /api/sleeper/link` at session establishment. The proof predicate is byte-identical (client-presented token, claim-match, live oracle); the server does not change at all. Everything else in the plan that survives contact with this correction is carried into the PRD: the problem statement (§1, fully verified), the banner/ESPN copy fixes (§2c), the unchanged-list (§2b's other items), risks 1/3, and the test-plan intent (§5, recast).

Collateral corrections:

- **Plan §3 "web/extension benefit for free"** — falls with the server helper. No capture flow → no credential to replay. The P2.5 known limitation stands; web verification remains top P3. Recorded as PRD non-goal R-9.
- **Plan §2a call-site design (sync-with-deadline at establishment + lazy gate backstop) and §7 timeout plumbing** — moot; no server helper. The client analogue is a single choke point in `auth.ts sessionInit()` (PRD R-2), which covers all establishment paths (sign-in, revalidate, league switch, ESPN init) with one hook.
- Had the server design been sound, it also contained a latent liveness bug worth recording: an establishment probe that missed its 3 s deadline would be memoized `inconclusive` with a ~5 min TTL, so the "lazy gate backstop" would no-op inside the TTL and the first gated call would 403 anyway. Only actual oracle outcomes should ever have been memoized. Academic now, but it reinforces that the design needed another pass.

## Q1 — Delete vs mark the stored credential on oracle-reject?

**Disposition: AGREE with "delete", recast to the client copy; server-side is moot.**

The server helper that would have deleted `sleeper_credentials` rows no longer exists, and this PRD forbids backend changes, so the question lands on the Keychain copy: **delete on definitive rejection** (`token_rejected`, `token_expired`, `token_user_mismatch`), retain on transport failure (PRD R-4). The Planner's justification transfers intact: `SleeperAuthError` fires only on Sleeper's explicit 401/403 ([sleeper_write.py:307-308](../../../../backend/sleeper_write.py)) — verified 2026-07-08 to trigger before query execution — so a reject is terminal for a 365-day token; matches propose's expired-token hygiene ([server.py:5156-5158]) and the `SleeperAuthError` docstring ("Caller should drop the stored token"). One caveat the plan didn't note: `_post_graphql` also classifies GraphQL-200 errors as auth by substring match ("auth"/"token"/"forbidden"/"login", [sleeper_write.py:319-325]) — a heuristic — but for a `__typename` no-op probe no plausible non-auth error matches those words, and the client-side consequence of a false delete is one extra manual capture, not data loss. The Planner's downgrade path (flappy-401 watch) is preserved: the failure mode is visible as repeat banner appearances.

Server hygiene footnote: on a rejected replay the *server's* stored credential (same JWT from the same capture) is very likely dead too, but the server already copes (propose maps `SleeperAuthError` to reconnect; expired creds are deleted at [server.py:5156-5158]). Left untouched per surgical-changes.

## Q2 — Probe `/api/session/ping` / revalidate paths too? Other mint paths?

## Planner review — round 2

*Planner agent, 2026-07-12. Method: re-derived the rebuttal's attack chain against the working tree independently, then adversarially reviewed PRD R-0..R-9 and the test plan. Verdict at bottom.*

### 1. The rebuttal — ACCEPTED IN FULL

For the log, explicitly: **the Planner accepts the rebuttal; plan §2a/§2b was unsound and the Author's security argument is correct.** Re-verified independently: `get_sleeper_credential` is keyed by `user_id` alone ([database.py:5776]); a squatter session mints from a public username ([server.py:8892-8923]) carrying the victim's `user_id`; `_maybe_autoverify` would therefore fetch the **victim's** live token, trivially pass its own claim re-check, and pass the oracle — verifying the attacker. My §2b parenthetical answered "can a squatter *store* a credential" when the load-bearing question was "does one *already exist* at the lookup key." The propose escalation is also correct and is the worst part: a verified squatter session plus the route's own credential pull ([server.py:5134-5153]) = real trades from the victim's account, strictly worse than pre-P1. The root-cause framing — *a server-stored credential can prove someone once controlled the account, never that the current session holder does* — is the right invariant, and R-0 codifies it correctly. I also accept the collateral corrections (web/extension "for free" claim falls; risk #4's reassurance was false) and the liveness-bug observation in my TTL memoization (correct: a deadline-missed probe memoized `inconclusive` would have made the "lazy backstop" no-op inside the TTL and 403'd the first gated call anyway).

### 2. The Keychain replay design — sound in principle, two blocking gaps

The corrected mechanism is right: possession replayed **by the client** through the existing hard-verified route preserves the P1 proof predicate byte-for-byte, and the attacker population that matters (remote username-knowers) has no path to the victim's Keychain. Token-at-rest is acceptable: the same JWT already sits server-side Fernet-encrypted and traversed this exact route at capture; iOS Keychain via expo-secure-store (already holding the session token, [client.ts]) is a narrower exposure than Sleeper's own `localStorage['token']`. Keychain survives app updates (the literal #126 trigger) and, in practice, iOS reinstalls; Android uninstall wipe is honestly recorded (R-9). Coverage claim verified: `sessionInit()` ([auth.ts:199-221]) is the single client function POSTing `/api/session/init`, and `initLeagueSession`, `useSession.revalidateSession` ([useSession.ts:244]), `switchLeague` ([useSession.ts:322]), and the ESPN body path all funnel through it — the choke point is real.

**BLOCKING B-1 — R-2.3's outcome mapping is not exhaustive, and there is no mobile unit-test harness to catch a misreading.** Unmapped today: **404 `feature_disabled`** (`/api/sleeper/link` is flag-gated on `trade.send_in_sleeper`, [server.py:4995-4996] — if the kill switch is ever flipped, every replay 404s), **401** (fresh token evicted mid-flight; note `client.ts:318-323` clears the stored session token on it), 503 `sleeper_unconfigured` (arguably "5xx" but name it), and any future/unknown status. A build agent reading "400/403 → delete" could defensibly delete the only proof copy on a 404. `mobile/package.json` has no jest — the PRD text is the sole defense. **Required fix:** restate R-2.3 as a closed rule — *delete ONLY on the three definitive codes (`token_expired`, `token_rejected`, `token_user_mismatch`); every other non-200 outcome, named or not, retains the token and returns `'inconclusive'`* — and add one line (R-8 or guardrails) recording that verification durability now inherits the `trade.send_in_sleeper` flag coupling (pre-existing for first capture; the replay extends it to re-verification).

**BLOCKING B-2 — cross-device revocation resurrection.** `POST /api/sleeper/link` **stores** the credential before the verified stamp ([server.py:5063-5072]) — replay and re-link are inseparable. So: user taps Disconnect on device A (`DELETE` → server credential removed, R-5 clears only *that device's* Keychain); device B's next session establishment is unverified → replay fires → **silently re-stores the revoked write credential server-side and re-arms `POST /api/trades/propose`.** The user's explicit revocation of the highest-blast-radius capability is undone by a background call with no user action. That is a consent regression the pre-fix world didn't have, and it rubs directly against "don't weaken the gate." **Required resolution (either):** (i) *recommended* — pre-check `GET /api/sleeper/link` (session-scoped, not read-gated, cheap DB read) and when `connected: false` skip the replay and clear the local copy — revocation stays sticky, and the #126 fix is untouched because the server credential persists across client reinstalls by construction (the expiry-deleted case is self-consistent: both copies share one `exp`, the replay would have 400'd anyway); or (ii) the operator explicitly signs off on "disconnect is per-device while a Keychain copy exists elsewhere" as a recorded limitation. Pick one in the PRD before build.

### 3. Non-blocking observations

- **N-1 (latency tail):** the awaited replay is bounded at 15 s (client deadline), not the plan's 3 s — a Sleeper hang adds up to 15 s to sign-in for token-holders. Rare and bounded; acceptable. Optional: race-cap the *await* at ~4 s and let the in-flight replay late-apply `setVerification` on success (the server session is stamped regardless; only the mirror lags).
- **N-2 (pre-init window, INIT-08):** sign-in mints the session at `extension/auth` ([auth.ts:28-33]) and the user can reach Main while `sessionInit` runs in the background ([auth.ts:239-243]) — gated reads in that window still 403, flipping the banner via the central listener and leaving TanStack queries in error until refetch. Pre-existing behavior, improved-not-eliminated by this PRD. Cheap tightening if desired: also fire the (single-flight) replay right after `signIn`'s `setSessionToken` — safe because verified state survives same-user re-init ([server.py:7279-7285]) — and/or invalidate errored queries on a `'verified'` outcome.
- **N-3 (mirror completeness):** on `'verified'`, mirror `user_verified: true, verified_via: 'sleeper'` too, not just `session_verified` — covers the linked-while-oracle-inconclusive user whose persisted marker was absent at init; `SleeperConnectScreen`'s own success mirror ([SleeperConnectScreen.tsx:76-81]) is the exact shape to reuse.
- **N-4 (one Keychain slot):** two FTF accounts alternating on one device overwrite each other's token → recapture ping-pong (correct, just frictional). Optional: key per user (`sleeper.link.jwt.<user_id>`; SecureStore accepts `[A-Za-z0-9._-]`, Sleeper ids are numeric).
- **N-5 (shared-device note):** R-5's retain-across-sign-out means anyone who signs in as that username *on that device* verifies. Same trust boundary as the original capture — fine — but add the sentence so it's a decision, not an accident.
- **N-6 (#114 flag):** device-side storage of the Sleeper credential should be glanced at by whoever owns the privacy docs; the server-side storage disclosure likely already covers it. No gate.
- **N-7 (test plan):** the five pytest sequence-pins are correct and well-targeted; add a sixth cheap pin for the 400 `token_expired` replay outcome (existing route behavior, pins B-1's delete trigger). The manual-operator-checkpoint rationale for Maestro is honest and I agree; if B-2 resolves via the GET pre-check, add a pin: disconnect → `GET` shows `connected: false` → (client contract) no replay.

### 4. Verdict

**NOT YET APPROVED — two blocking items (B-1, B-2), both resolvable with PRD edits; no design change required.** The rebuttal is accepted for the record; the replacement architecture (R-0 invariant, client-side persistence, replay through the existing route, zero backend app-code change) is endorsed and should proceed once B-1's closed outcome rule and B-2's revocation decision are written into the PRD.

### B-2 — CLOSED (PRD R-2.2), per orchestrator arbitration: pre-check over operator sign-off

Accepted without reservation — the Planner caught a real consent regression the Author missed: since `POST /api/sleeper/link` stores before stamping ([server.py:5063-5072]), replay and re-link are inseparable, and a background replay from device B would resurrect a credential the user explicitly revoked on device A, re-arming propose. New R-2.2: before any replay, `GET /api/sleeper/link` (existing `getSleeperLinkStatus()`); `connected: false` → skip the replay **and** delete the local copy (revocation sticky everywhere — R-5 updated to say other devices self-neutralize); `connected: true` → proceed; **any pre-check failure (404/401/transport/unknown) → skip + retain + `'inconclusive'`** — the token is never POSTed without a confirmed live server-side link, which also folds the B-1 flag-off case in safely. The Planner's self-consistency note holds: the server credential survives client reinstalls by construction, so the pre-check never blocks the #126 fix; the `expired: true` flag (server's copy) deliberately does not short-circuit — R-2.4 owns every replay outcome. Success criterion 4 (revocation stickiness) and a guardrail added; pytest pin added (below).

### Non-blocking dispositions

- **N-1 (15 s await tail) — ACCEPTED.** Primary call site now races the single-flight replay against a 4 s cap; on a late `'verified'` the mirror is late-applied under a guard (only if `verification` still shows `session_verified: false` for the same signed-in user — never clobber newer state). Added wait is hard-capped; the server session is stamped by the replay regardless, only the mirror could lag.
- **N-2 (INIT-08 pre-init window) — ACCEPTED, first half.** Fire-and-forget warm-up call right after `signIn()`'s `setSessionToken` ([auth.ts:26-35]); single-flight makes the later awaited race resolve instantly. The "invalidate errored queries on `'verified'`" suggestion is **REJECTED** for V1: the warm-up shrinks the window to roughly one RTT, TanStack retry/refocus/navigation already recover errored queries, and wiring a query-client dependency into `sendInSleeper.ts` (or a new listener) is cross-layer churn out of proportion to a transient sub-second window. Revisit only if the operator checkpoint actually observes stuck errored screens.
- **N-3 (mirror completeness) — ACCEPTED.** On `'verified'` the mirror is the full SleeperConnectScreen success shape ([SleeperConnectScreen.tsx:76-81]): `session_verified: true, user_verified: true, verified_via: 'sleeper'`, `enforced` from the init response — covers the linked-while-oracle-inconclusive user whose persisted marker was absent at init.

Pin 7: link+verify → `DELETE` → `GET` returns `connected: false` (the exact server read B-2's pre-check depends on). Client-side halves of both remain contract-only, consistent with the no-mobile-harness reality the Planner flagged in B-1.
