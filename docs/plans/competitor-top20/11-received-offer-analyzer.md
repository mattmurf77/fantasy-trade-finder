# 11. Received-offer analyzer (read-only inbox)

> Tier 1 · #11 · NEW · Effort L · Sources: DDr (offer inbox precedent), FTF (read-only policy line)

## Summary

"Should I take this deal?" is the highest-intent moment in dynasty fantasy, and FTF is currently absent from it. DynastyDealer proves the demand: it renders real pending Sleeper offers in-app with ACCEPT / MODIFY / DECLINE buttons — but it does so via user auth tokens (its 2FA sign-in stores Sleeper tokens locally and *writes* into Sleeper), which is exactly the ToS exposure FTF has ruled out (PASS items #79/#80: "analyze everything, execute nothing"). FTF builds the *analysis* half: every pending offer the user receives gets a v2-engine workup — verdict + gap (#6), both-team impact (#5), and counter suggestions via the swap machinery (#3) — and the only action buttons are deep links back to Sleeper, where the user accepts or declines themselves.

The hard unknown is ingestion. Sleeper's public read API serves rosters, users, and *completed* transactions; whether **pending** trade offers are readable without an auth token is unverified and is precisely what the #83 Sleeper-auth feasibility memo must answer before V2 scoping. The plan therefore ships in two phases: **V1 — manual offer entry** (user picks the league, the offering team, and the assets on each side; the analyzer does the rest — zero Sleeper unknowns, full analysis value), and **V2 — automatic inbox** (offers appear without typing) only if #83 clears a read path FTF is comfortable with. V1 is not a throwaway: it's the same analyzer screen with a different input method, and it doubles as the analyzer for offers received via text/league chat, which no inbox integration would catch anyway.

## PRD

### Problem & user story

> As a user who just got a trade offer in Sleeper, I want FTF's read on it — who wins, by how much, what it does to both rosters, and what counter I should send — before I touch accept or decline.

Today the user's options are eyeballing it or rebuilding the offer in a competitor's referee-style calculator. FTF has strictly better inputs (their personal Elo, league-mate rankings, roster needs) and stronger machinery (v2 mutual-gain valuation, sweetener search), but no doorway for an inbound offer.

### Goals / Non-goals

**Goals**
- Analyze any received offer through the v2 engine: verdict banner (#6 object), both-team impact (#5 panel), and 1–3 suggested counters (#3 swap machinery / v3 sweetener pass).
- V1: manual entry flow good enough to reconstruct an offer in under 60 seconds (roster-aware pickers, not free-text search).
- V2 (gated on #83): pending offers listed automatically, marked analyzed/unanalyzed.
- Every action affordance deep-links to Sleeper (#12's link work). FTF never accepts, declines, or counters *in* Sleeper.

**Non-goals**
- No write actions, ever — no accept/decline/counter API calls, no auth-token collection for write purposes. This is policy, not backlog (PASS #79/#80).
- No offer *sending* (that's #12's deep link).
- No notification on new offers in V1 (V2 + #18 can add "you got an offer — here's our read" later; it depends on a polling read path #83 must validate).

### Functional requirements

- **FR1** New analyzer surface (web page + mobile screen): input = league + counterparty + give/receive asset lists; output = full analysis card.
- **FR2** V1 manual entry: user selects league (from synced leagues), then the offering team (league members already synced), then taps assets from the two real rosters (players + picks). No free-text entry of player names.
- **FR3** Analysis output: (a) #6 verdict object rendered with received-offer copy variants ("They're asking for ~480 too much — counter"); (b) both-team before/after impact (#5); (c) up to `counter_max` (model_config, default 3) counter suggestions generated by re-running the package through the swap/sweetener machinery with the mutual-gain gate.
- **FR4** Each analysis is persisted (see schema) so the user can revisit, and so engine telemetry can compare analyzer verdicts to eventual outcomes.
- **FR5** Action row: "Open in Sleeper" deep link (#12) + "Build a counter" (opens #3's swap builder seeded with the offer). No accept/decline controls.
- **FR6** V2 (flag-gated, post-#83): `GET /api/offers` lists pending received offers per league with analysis status; selecting one pre-fills FR1's input. Ingestion mechanism = whatever #83 approves; if #83 finds only token-based access, V2 is **not built** and V1 remains the product.
- **FR7** Outcome capture: user can mark an analyzed offer "accepted / declined / countered" manually (V1); V2 may reconcile automatically against Sleeper's completed-transactions read API (completed trades are public-read — verify exact endpoint shape in #83's memo rather than assuming).

### UX notes (per client)

- **Mobile** (`mobile/src/screens/`): primary surface — offers arrive on phones. New `OfferAnalyzerScreen` reached from the Trades tab ("Got an offer?") and registered in `mobile/src/navigation/`. Entry flow is three steps (league → from-team → assets), each a tappable list, then a single analysis card using the existing TradeCard visual language.
- **Web**: page linked from the trades view; same three-step entry, side-by-side layout.
- Verdict copy uses received-offer variants of the #6 matrix — the "favored" perspective flips naturally (the engine scores from the user's side already), but action copy changes: "counter with…" instead of "ask for…".
- Empty state teaches the boundary explicitly: "FTF analyzes offers; you accept or decline in Sleeper."

### Success metrics

- Analyses per active user per week (event `offer_analyzed`).
- Counter-suggestion engagement: % of analyses where a counter is opened in the swap builder.
- Outcome-marked rate and verdict calibration: of offers FTF called "lopsided against you," what fraction did users decline? (engine-metrics dashboard, #84.)

### Acceptance criteria

- [ ] V1: an offer reconstructed manually produces verdict + both-team impact + ≥1 counter (when one exists) in a single analysis card, on web and mobile.
- [ ] Asset pickers show only the two real rosters (players + picks) for the chosen league.
- [ ] No code path performs an authenticated Sleeper write; no Sleeper credentials/tokens are collected (review checklist item).
- [ ] Analyses persist and are listed under "Past offers" with manual outcome marking.
- [ ] V2 work does not start until #83's memo is filed and its recommendation accepted by the operator.
- [ ] docs updated: api-reference.md (new routes), data-dictionary.md (new table), glossary.md ("received offer", "counter suggestion").

## HLD

### Components touched

- `backend/server.py` — new `/api/offers/*` route family; reuses session auth.
- `backend/trade_service.py` — exposed scoring entry point shared with #3's rescore (`POST /api/trades/rescore` — referenced, not redesigned here): given explicit give/receive + league + counterparty, return v2 valuation, verdict inputs, impact data. Counter generation = constrained re-run of the package search seeded with the offer's assets.
- `backend/database.py` — `received_offers` table.
- `web/` new page + `mobile/src/screens/OfferAnalyzerScreen` + navigation registration.

### Data flow

V1: client posts the reconstructed offer → server loads both rosters + the user's shrunk Elo values (+ counterparty's real rankings when they exist, else consensus basis — same rule as `/api/trades/generate`) → scores package both directions → persists analysis row → returns analysis card. Counters run the swap/sweetener search server-side in the same request (bounded; this is one pair, not a league scan).

V2 (post-#83): a fetch step populates pending offers per league via whatever read mechanism #83 approves; everything downstream is identical.

### Flags & config interplay

- Flag `offers.analyzer` (default false) gates the whole surface; `offers.inbox_auto` (default false, **must stay false until #83 memo accepted**) gates V2 ingestion.
- `model_config`: `counter_max` (3), `offer_analysis_ttl_days` (retention for stale rows).
- Consumes #6 verdict thresholds and #5 impact computation; both must exist first.

## LLD

### API changes (routes + example payloads)

```
POST /api/offers/analyze
{
  "league_id": "112233...",
  "from_user_id": "55667...",          // offering league-mate
  "give":    ["8136", "pick:2027:2"],  // assets the user would send
  "receive": ["4046"],
  "source": "manual"                    // V2: "sleeper_inbox"
}
→ 200 {
  "offer_id": 41,
  "verdict": { "band": "slight", "favored": "them", "gap_value": 480, ... },
  "impact":  { "you": {...}, "them": {...} },          // #5 shape
  "counters": [ { "give": [...], "receive": [...], "verdict": {...} } ],
  "sleeper_url": "https://sleeper.com/leagues/112233.../..."   // from #12; null until its spike lands
}

GET  /api/offers                  → list persisted analyses (?league_id=)
POST /api/offers/<id>/outcome     → { "outcome": "accepted"|"declined"|"countered"|"expired" }
```

V2 adds `GET /api/offers/pending?league_id=` — shape deliberately unspecified until #83.

### Schema changes (SQLAlchemy Core, SQLite + Postgres)

```python
received_offers_table = Table("received_offers", metadata,
    Column("id",            Integer, primary_key=True, autoincrement=True),
    Column("user_id",       String,  nullable=False),
    Column("league_id",     String,  nullable=False),
    Column("from_user_id",  String),
    Column("give_json",     String,  nullable=False),   # asset id list
    Column("receive_json",  String,  nullable=False),
    Column("analysis_json", String),                    # verdict+impact+counters snapshot
    Column("source",        String,  nullable=False),   # 'manual' | 'sleeper_inbox'
    Column("outcome",       String),                    # accepted|declined|countered|expired|null
    Column("created_at",    String,  nullable=False),
    Index("ix_received_offers_user", "user_id", "created_at"),
)
```

Added via the existing `_migrate_db()` idempotent pattern; JSON-as-text columns work on both SQLite and Postgres. Update data-dictionary.md.

### Sleeper integration notes (read-only boundary)

- **V1 touches Sleeper not at all** beyond data FTF already syncs (rosters, members, picks).
- **Pending-offer readability is UNKNOWN.** Sleeper's documented public read API covers users/leagues/rosters/transactions; whether pending (un-executed) trade offers are exposed without an auth token is the central question the **#83 feasibility memo** exists to answer, including the ToS posture of any token-based read. Nothing in this plan asserts it works. If #83's answer is "token-only," FTF's answer is "then V1 is the product" — DynastyDealer's token approach is the documented anti-pattern, not a fallback.
- Completed-transaction reads (for FR7 auto-outcome) are *believed* public but the exact endpoint/shape goes in #83's memo too.
- No accept/decline/counter writes under any finding. Deep links (#12) carry all action.

### Rollout

Flag `offers.analyzer` default **false** → enable for operator account → beta cohort → on. `offers.inbox_auto` stays false indefinitely pending #83 + operator sign-off. V1 has no new external dependencies, so rollout risk is engine-load only (one extra scoring path, single-pair scope).

### Open questions

1. #83 outcome — the entire V1/V2 fork. Memo must cover: pending-offer visibility, transaction-read shape, polling etiquette/rate limits, ToS posture.
2. Should counters respect the mutual-gain gate strictly, or relax to "fair-by-consensus" when the counterparty has no rankings? (Proposal: same `basis` rules as `/api/trades/generate`.)
3. Pick entry UX in manual mode — picks lack stable display names; reuse `/api/league/picks` inventory as the picker source.
4. Does analyzing an offer feed `trade_impressions`/preference signals? (Likely yes — an offer the user declines after a "lopsided" verdict is a strong label — but decide with #7's schema.)
